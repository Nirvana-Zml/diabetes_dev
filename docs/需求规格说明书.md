## 1. 引言

### 1.1 编写目的

本文档旨在为**糖尿病智能助手**系统建立完整、规范的需求规格说明，作为项目开发、测试、验收的共同基准。具体目标如下：

1. **定义功能边界**：明确系统"做什么"与"不做什么"，避免需求蔓延；
2. **统一各方理解**：为前端、后端、AI 工程、测试、产品团队提供一致的需求认知；
3. **指导测试设计**：为功能测试、性能测试、安全测试和 AI 准确性评估提供可验证的验收标准；
4. **支撑项目管理**：为迭代计划、风险识别和资源分配提供输入；
5. **满足合规要求**：作为医疗健康类软件的需求档案，符合文档规范化要求。

### 1.2 项目背景与系统定位

#### 1.2.1 项目背景

糖尿病已成为全球性公共卫生挑战。世界卫生组织数据显示，全球糖尿病患者数量已达数亿并持续攀升。糖尿病引发的并发症严重威胁患者健康，同时给社会和家庭带来沉重经济负担。在此背景下，如何借助人工智能技术实现糖尿病的早期预警、科学预防和个性化管理，成为医疗健康领域的重要课题。

本项目依托国家推进"健康中国2030"建设、加强慢性病管理的政策导向，融合 **Dify 智能体编排平台**与 **DeepSeek 大模型**，构建面向糖尿病患者及高危人群的预治一体化智能管理平台。

#### 1.2.2 系统定位

本系统定位为**基于 LLM 和智能体技术的糖尿病预治一体化管理平台**，核心价值主张：

- **智能驱动**：以 DeepSeek 大模型为核心推理引擎，实现风险预测、方案生成、AI 问诊等复杂能力；
- **知识加持**：以 Milvus 权威医学知识库为数据底座，通过 RAG 技术确保 AI 输出的专业性与合规性；
- **全链路覆盖**：从健康科普 → 风险筛查 → 在线咨询 → 方案制定 → 行为打卡 → 效果追踪 → 主动干预，形成患者自我管理的完整闭环；
- **私有化部署**：支持本地/私有云部署，保障医疗数据安全与隐私合规。

## 2. 总体描述

### 2.1 产品视角

**糖尿病智能助手**是一套基于微服务架构的 Web 应用系统，面向糖尿病患者、高危人群、内分泌科医生、健康管理师及系统管理员，提供"科普 → 评估 → 咨询 → 方案 → 打卡 → 分析 → 推荐 → 干预"的完整健康管理闭环。

系统由以下核心组件构成：

| 组件 | 技术实现 | 用户群体 |
|------|----------|----------|
| 用户端前端 | Vue 3 SPA | 普通用户 |
| 管理端前端 | Vue 3 SPA | 管理员 |
| 微服务后端集群 | Spring Cloud 8 微服务 + 1 网关 | — |
| AI 编排层 | Dify + DeepSeek + Milvus | — |
| 数据存储层 | MySQL + Redis + MinIO + Milvus + KingbaseES | — |

### 2.2 用户角色

| 角色 | 标识 | 主要职责与权限 |
|------|------|----------------|
| **普通用户** | `user` | 浏览科普内容、AI 科普问答、完成风险评估、发起在线咨询、查看/生成健康方案、每日打卡、浏览资讯、管理个人资料、管理健康档案、管理消息通知、数据导出 |
| **医生** | `doctor` | 管理问诊会话、查看患者健康档案（需授权）、回复患者咨询、获取 AI 辅助建议 |
| **管理员** | `admin` | 资讯创建/审核/发布/管理、科普视频上传/管理、统计看板查看、审计日志查询/导出、用户管理 |

### 2.4 设计与实现约束

| 约束类别 | 约束内容 |
|----------|----------|
| **架构约束** | 采用 Spring Cloud 微服务架构，按业务域拆分 8 个独立服务；API 网关统一路由；数据库隔离 |
| **技术栈约束** | Java 17 + Spring Boot 3.3 + MyBatis；Vue 3 + Element Plus + Vite；Dify 编排 + DeepSeek LLM |
| **部署约束** | Docker Compose 容器化部署；支持私有化部署；医疗数据不出私有网络 |
| **法律合规约束** | 遵循《个人信息保护法》《数据安全法》；AI 输出强制附加免责声明；不替代医生诊断 |
| **安全约束** | 全链路 HTTPS/TLS 1.3；JWT 身份认证；RBAC 三级权限；AES-256 加密敏感字段 |
| **性能约束** | 普通页面 ≤ 2s；AI 首 Token ≤ 3s；支持 ≥ 100 并发用户 |
| **数据约束** | 用户健康数据加密存储；数据保留最近 1 年活跃记录；支持数据导出（PDF/Excel） |

## 3. 功能需求

> 本章按模块逐一描述功能需求。优先级标注：**P0**=核心必达（MVP）、**P1**=重要（二期）、**P2**=增强（三期）、**P3**=远期规划。

### 3.1 用户认证与账户管理

| 编号 | 功能需求 | 优先级 | 详细说明 |
|------|----------|--------|----------|
| FR-AUTH-01 | 手机号+验证码注册 | P0 | 用户通过手机号获取短信验证码完成注册；验证码 60 秒发送间隔限制；验证码 Redis 存储，5 分钟过期 |
| FR-AUTH-02 | 手机号+验证码登录 | P0 | 支持手机号+验证码登录；登录成功返回 JWT access_token（2h 有效）与 refresh_token（7 天有效） |
| FR-AUTH-03 | Token 自动刷新 | P0 | access_token 过期后可通过 refresh_token 获取新 token |
| FR-AUTH-04 | 密码修改 | P1 | 用户可设置/修改登录密码；密码 bcrypt 加密存储 |
| FR-AUTH-05 | 个人资料管理 | P0 | 支持修改昵称、头像（上传至 MinIO）、性别、出生日期等 |
| FR-AUTH-06 | 账户安全 | P1 | 支持绑定/更换手机号、绑定/更换邮箱；操作需验证码二次确认 |

### 3.2 科普展示首页与 AI 问答

| 编号 | 功能需求 | 优先级 | 详细说明 |
|------|----------|--------|----------|
| FR-HOME-01 | 首页内容展示 | P0 | 展示轮播 Banner、知识分类卡片（糖尿病基础/饮食/运动/用药/并发症 5 大分类）、热门科普视频 |
| FR-HOME-02 | AI 科普问答 | P0 | 右下角悬浮「AI 科普助手」按钮，点击展开对话窗口；用户自然语言提问，SSE 流式返回 Markdown 回答 |
| FR-HOME-03 | RAG 知识增强 | P0 | 后端通过 Milvus 检索 Top-5 相关文档片段，注入 Dify 问答工作流作为知识上下文 |
| FR-HOME-04 | 引用来源展示 | P1 | AI 回答附带引用来源（文档名+相似度），增强可信度 |
| FR-HOME-05 | 多轮对话 | P1 | 支持多轮对话（`conversation_id`），保留最近 10 轮对话上下文 |
| FR-HOME-06 | 对话历史查询 | P2 | 用户可查看历史问答记录 |
| FR-HOME-07 | 语音输入（STT） | P1 | 用户通过麦克风录音 → 后端调用 Dify STT 工作流识别为文字 → 自动填入输入框或直接提问；录音时长 ≤ 60 秒；不支持 WebM（仅 MP3/M4A/WAV/AMR） |
| FR-HOME-08 | 科普视频浏览 | P1 | 支持播放科普视频，含封面图、视频时长、播放控制 |
| FR-HOME-09 | 安全过滤 | P0 | 所有 AI 回答末尾固定附加免责声明：「以上内容仅供参考，不构成诊疗建议」；超出科普范围的问题（要求开处方/诊断）婉拒并引导在线问诊 |

### 3.3 糖尿病风险预测

### 7.2 RAG 检索增强生成

| 需求项 | 说明 |
|--------|------|
| 知识库数据源 | DiaKG 41 篇诊疗指南 JSON（约 2292 段、28 万字），转换为纯文本后分段入库 |
| 分段策略 | 按文档内顺序合并至 400~600 字/chunk，重叠 80 字 |
| Embedding 模型 | Qwen3-Embedding 0.6B / BAAI bge-large-zh-v1.5（1024 维） |
| 检索流程 | 用户查询 → Embedding 向量化 → Milvus ANN 检索 Top-K → 相似度过滤（≥0.7） → 拼装 `knowledge_context` → 注入 Dify |
| 降级策略 | Milvus 不可用或检索无结果时，`knowledge_context=""`，LLM 基于通用知识回答，标注「基于通用医学知识」 |
| 应用场景 | 科普问答、AI 问诊（诊疗指南+药物信息）、方案生成（饮食/运动指南）、画像 Agent（医学溯源）、资讯推荐（语义检索+重排） |

### 7.3 Prompt 工程要求

| 场景 | 关键 Prompt 要求 |
|------|-----------------|
| 科普问答 | 以「小糖助手」角色回答；优先基于 `knowledge_context`；Markdown 排版；≤ 500 字；末尾固定免责声明；拒答处方/诊断请求 |
| AI 问诊 | 以指定医生角色直接回复患者；基于患者档案、历史对话、知识库生成建议；输出 `content`（Markdown）+ `suggestion`（结构化辅助）；末尾免责声明 |
| 风险评估 | 以风险评估专家角色分析；输出严格 JSON（risk_level/risk_score/factors/suggestions/confidence）；标注置信度 |
| 方案生成 | 以营养师+运动康复师角色；基于用户画像+每日热量+知识库生成四维方案（饮食/运动/作息/用药）；末尾「请在医生指导下执行」 |
| 打卡行为分析 | 分析行为规律、异常检测、生成改善建议；输出结构化 JSON（summary+behavior_patterns+anomalies+improvements） |

### 7.4 模型选型与参数

| 参数 | 通用推理（V3） | 深度推理（R1） | 说明 |
|------|---------------|---------------|------|
| temperature | 0.7 | 0.3 | 低温度保证推理一致性 |
| top_p | 0.9 | 0.9 | 核采样 |
| max_tokens | 2048（动态） | 4096（动态） | 根据场景调整 |
| stream | true | true | 统一启用流式 |
| timeout | 30s | 60s | R1 推理时间更长 |

模型选择策略：

- 科普问答、AI 问诊、方案生成、资讯草稿、打卡分析 → DeepSeek-V3（通用推理）
- 糖尿病风险评估、复杂病例分析 → DeepSeek-R1（深度推理）

### 7.5 降级策略

| 场景 | 降级方式 |
|------|----------|
| 未配置 Dify API Key | 返回本地规则/模板结果，标注降级模式 |
| Dify 调用失败/超时 | 自动重试 3 次，指数退避（1s→2s→4s）；最终失败切换本地降级 |
| Milvus 不可用 | `knowledge_context=""`，基于 LLM 通用知识回答，标注来源 |
| 知识库检索无结果 | 同上，LLM 标注「基于通用医学知识」 |
| DeepSeek-R1 超时 | 降级使用 DeepSeek-V3 快速推理，标注"快速评估模式" |

| 验收项 | 验收条件 |
|--------|----------|

| 普通用户请求管理端接口 | 返回 403 Forbidden |
| SQL 注入测试 | 参数化查询防护，注入无效 |

| 接口频率限制 | 高频调用触发 429 Too Many Requests |

| 数据加密 | 数据库直接查看敏感字段为密文 |
