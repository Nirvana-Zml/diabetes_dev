#### c. 项目产生背景

在当今全球化的时代，糖尿病已然成为一个令人忧虑的全球性健康难题。随着生活方式的急剧转变——高糖高脂饮食的普遍化、运动量的大幅减少，以及人口老龄化进程的加速，糖尿病的发病率正以惊人的速度持续攀升。世界卫生组织的数据显示，全球糖尿病患者数量已达数亿之多，并且这一数字仍在不断增加。

糖尿病不仅会引发各种严重的并发症（如心血管疾病、肾病、失明等），严重威胁患者的生命健康和生活质量，还会给社会和家庭带来沉重的经济负担，包括高额的医疗费用、长期的护理成本以及生产力的损失。因此，如何科学有效地预防和治疗糖尿病，成为了全球公众和医疗界共同关注的焦点问题。

在此背景下，项目依托国家推进"健康中国2030"建设、加强慢性病管理的政策导向，创新性地融合DeepSeek和Dify的前沿智能体开发技术，精心构建一个一站式的糖尿病预治管理体系，助力提升糖尿病预防和治疗水平，推动糖尿病防治工作的科学化、规范化发展。

## 3．对现有系统的分析

这里的现有系统是指当前实际使用的系统，可能是计算机系统、机械系统或人工系统。分析现有系统的目的是为了进一步阐明开发新系统或修改现有系统的必要性。

### 3.1 处理流程和数据流程

现有系统（传统人工糖尿病健康管理方式）的基本处理流程如下：

```
┌─────────────────────────────────────────────────────────────────────┐
│                       传统人工糖尿病健康管理流程                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────────┐  │
│  │ 患者自查  │───▶│ 医院挂号  │───▶│ 排队候诊  │───▶│ 医生面诊评估  │  │
│  │ 或被动就医 │    │ （窗口/APP）│    │ （等待时间  │    │ （问诊+检查开单）│  │
│  └──────────┘    └──────────┘    │   不确定）  │    └──────┬───────┘  │
│                                  └──────────┘           │         │
│                                                          ▼         │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────────┐  │

│  │              方案制定：营养师/健康管理师人工沟通 → 1~3工作日     │  │
│  │              行为记录：手动笔记 → 缺乏统计分析与趋势追踪         │  │
│  └──────────────────────────────────────────────────────────────┘  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**现有系统数据流程说明：**

1. **风险评估数据流**：患者健康指标 → 手动填写体检表 → 医生人工评估 → 口头/纸质传达结果 → 数据无系统化归档
2. **问诊咨询数据流**：患者症状描述 → 医生听诊结合经验判断 → 纸质病历记录 → 下次就诊时凭记忆或翻阅历史病历
3. **健康管理数据流**：患者自行记录（纸质/手机备忘录）→ 无统一格式 → 复查时提供给医生参考 → 医生凭经验调整方案
4. **知识获取数据流**：用户搜索 → 信息碎片化、来源不明 → 自行判断真伪 → 可能采纳错误信息

### 3.2 工作负荷

现有系统（传统人工方式）的主要工作负荷如下：

| 工作类型               | 负荷描述                                                                     | 工作量估计                       |
| ---------------------- | ---------------------------------------------------------------------------- | -------------------------------- |
| **糖尿病风险评估**     | 需患者到院 → 挂号 → 排队 → 面诊 → 开检查单 → 完成多项检查 → 等待结果 → 复诊 | 每次评估耗时数小时至数天         |
| **个性化方案制定**     | 需营养师/健康管理师与患者多次沟通，查阅资料，手动制定方案                     | 每例方案耗时1~3个工作日          |
| **健康知识科普**       | 依赖患者自行搜索或参加线下讲座                                               | 信息碎片化，获取效率低           |
| **健康行为记录**       | 患者手动记录饮食、运动、用药、血糖等数据                                     | 每日耗时10~20分钟，易遗漏       |
| **数据统计与分析**     | 手动汇总历史记录，人工绘制趋势图表                                           | 每次分析耗时30分钟~1小时         |
| **医患沟通**           | 仅限线下门诊时间，受地域和排班限制                                           | 每次沟通需往返医院，平均耗时2~4小时 |
| **患者档案管理**       | 纸质病历或分散的电子表格，缺乏统一管理                                       | 查找历史数据耗时10~30分钟/次    |
| **健康资讯获取**       | 患者自行搜索网络信息，信息来源不明，准确性无保障                             | 筛选有效信息耗时不确定           |

### 3.6 局限性

现有系统（传统人工糖尿病管理方式）存在的主要局限性如下：

| 序号 | 局限性                   | 具体说明                                                                                                                                     |
| ---- | ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------- |
| 1    | **处理时间慢**           | 风险评估需患者多次往返医院，从挂号到获取完整评估结果通常需数小时至数天，无法做到及时预警和早期干预。                                            |
| 2    | **响应不及时**           | 患者突发健康疑虑时，无法即时获得专业解答；医患沟通仅在门诊时间内进行，缺乏实时在线的问诊渠道。                                                  |
| 3    | **数据存储能力不足**     | 患者健康数据以纸质病历或分散电子表格存储，缺乏系统化的数据库管理，数据丢失、难以检索和统计分析的问题严重。                                        |
| 4    | **处理功能不够**         | 缺乏自动化的风险评估模型，依赖医生个人经验判断，评估结果的一致性和客观性难以保证；无法实现基于大数据的个性化方案推荐。                              |
| 5    | **信息碎片化严重**       | 患者健康档案、就诊记录、检查报告分散在不同医院和科室，无法形成完整的个人健康画像，医生难以全面了解患者的历史健康趋势。                              |
| 6    | **知识获取不可靠**       | 患者自行搜索的健康信息质量参差不齐，缺乏权威验证，存在被虚假医疗信息误导的风险。                                                                |
| 7    | **行为管理缺失**         | 缺乏系统化的饮食、运动、用药行为记录与追踪机制，患者健康行为依从性难以量化和监督，行为改善缺乏数据驱动的正向激励。                                |
| 8    | **人力资源效率低**       | 医生和营养师花费大量时间在重复性工作上（如基础指标计算、常规方案制定），无法将有限的专业精力集中在疑难病例的判断和患者深度沟通上。                |
| 9    | **地域和时效局限**       | 患者受限于地理位置和门诊时间，无法实现随时随地的健康管理服务触达，基层和偏远地区的优质医疗资源可及性更差。                                          |
| 10   | **数据准确性无保障**     | 人工记录存在漏记、错记风险；手工汇总统计容易出错，数据追溯和审计困难。                                                                          |

**为什么对现有系统的改进性维护已经不能解决问题：**

传统人工管理方式的局限性是结构性、系统性的问题，而非局部功能不足。简单地在现有纸质/人工流程上叠加电子表格或简单的信息录入系统，并不能从根本上解决以下核心矛盾：

- **效率矛盾**：专业医疗人力资源有限 vs 日益增长的糖尿病患者管理需求
- **时空矛盾**：线下门诊的地域和时间限制 vs 患者随时随地的健康管理需求
- **知识矛盾**：海量碎片化的网络健康信息 vs 患者对权威、个性化健康指导的需求
- **数据矛盾**：分散、非结构化的健康数据存储 vs 基于完整健康画像的精准干预需求
- **管理矛盾**：患者自我管理缺乏工具和激励 vs 慢性病管理需要长期行为改变

因此，需要开发一个全新的、基于AI技术的智能化糖尿病预治管理系统，以体系化的方式解决上述根本矛盾。

## 4．所建议技术可行性分析

### 4.1 对系统的简要描述

"糖尿病预治智能助手"是一套基于前沿AI技术的糖尿病预治智能管理平台。系统采用四层架构设计，以DeepSeek大模型为核心推理引擎，Dify智能体平台为中枢编排引擎，结合Milvus向量知识库、MySQL业务数据库、Redis缓存和MinIO对象存储，构建完整的"前端交互层 → Dify智能体层 → DeepSeek大模型层 → 医学知识库层"技术栈。

## 7．其他可供选择的方案

### 7.1 可选择的系统方案1：购买商业糖尿病管理SaaS服务

**方案描述：**

直接采购市面上已有的商业糖尿病管理SaaS平台（如智糖AI、微糖、糖护士等商业产品），通过订阅模式使用其提供的在线服务功能。

**方案简述（参照第4章提纲）：**

| 对比维度           | 商业SaaS方案                                                             |
| ------------------ | ------------------------------------------------------------------------ |
| **系统描述**       | 第三方提供的标准化糖尿病管理云平台                                        |
| **处理流程**       | 标准化的数据录入 → 规则引擎评估 → 模板化方案生成                           |
| **技术先进性**     | 通常使用传统规则引擎或较早期AI模型，定制化能力有限                         |
| **数据安全性**     | 数据存储于第三方服务器，存在数据主权和隐私泄漏风险                          |
| **成本**           | 按年/按用户收费，通常¥5~20万/年（以服务1000用户计）                        |

**未被选中的理由：**

1. **教学适配性差**：商业产品的技术架构为"黑盒"，学生无法深入理解和实践AI智能体开发、工作流编排、大模型应用等核心技术，不能作为Web前端设计与开发课程的实训项目。
2. **定制化能力弱**：商业产品的功能模块固定，无法根据教学需求和特定场景进行灵活调整和二次开发。
3. **数据主权风险**：用户健康数据需上传至第三方平台，不符合医疗数据本地化管理和私有化部署的安全要求。
4. **长期成本高**：按订阅制收费，3年累计成本可能超过¥15~60万，远高于自主开发方案的一次性投入+低运维成本。
5. **技术依赖性**：对第三方服务商的持续经营能力和服务水平存在依赖性，存在服务中断或终止的风险。

### 7.2 可选择的系统方案2：基于其他大模型平台的自主开发

**方案描述：**

采用其他大语言模型（如OpenAI GPT-4、百度文心一言、阿里通义千问、讯飞星火等）替代DeepSeek，结合LangChain或Coze等AI编排框架替代Dify进行自主开发。

**方案简述（参照第4章提纲）：**

| 对比维度           | 其他大模型方案                                                                                                       |
| ------------------ | -------------------------------------------------------------------------------------------------------------------- |
| **系统描述**       | 基于GPT-4/文心一言/通义千问等第三方大模型 + LangChain/Coze编排框架的糖尿病管理平台                                      |
| **处理流程**       | 类似四层架构，但模型层使用不同的LLM，编排层使用不同的框架                                                               |
| **技术先进性**     | GPT-4能力领先但成本高；国产模型能力参差不齐；LangChain灵活性高但开发门槛更高                                             |
| **成本**           | GPT-4 API费用高昂（约¥0.03~0.06/1K tokens）；文心一言/通义千问有免费额度但超出后收费                                      |

**未被选中的理由：**

1. **成本优势不及DeepSeek方案**：DeepSeek当前完全免费商用，而GPT-4 API调用费用高昂（预估年费¥5~20万），其他国产模型虽有免费额度但超量后收费，长期运营成本不可控。
2. **编排平台易用性不及Dify**：Dify提供可视化拖拽式工作流构建，开发门槛低，非常适合教学环境；LangChain需大量代码编写，对学生的Python基础和AI工程能力要求更高，不利于聚焦前端开发的教学目标。
3. **中文医学知识理解能力**：DeepSeek在中文语义理解和医学领域知识覆盖上表现优异，其128K上下文窗口能力也优于多数竞品，更适合处理复杂的医学知识推理场景。
4. **私有化部署兼容性**：DeepSeek + Dify均支持私有化部署和API直连，在数据安全和合规性方面具有一致的优势。部分商业模型不支持完全的私有化部署。
5. **教学契合度**：DeepSeek + Dify + Cline + VS Code构成完整的"AI辅助开发"教学技术栈，高度契合项目的教学特色定位，其他方案无法提供一致的AI教学体验。

> **说明**：经过充分比较和论证，本项目未发现除上述两个方案之外的其他需重点考虑的可选择方案。当前推荐的DeepSeek + Dify方案在技术先进性和经济可行性方面均优于其他方案。
