## 1. 背景与定位

### 1.1 变更说明

原「用药提醒」独立模块方案已**取消**。用药相关能力保留在**生活打卡 → 用药 Tab**（事后记录），不再单独做「到点提醒服药 + 确认已服」闭环。

本模块统一承接 **打卡提醒** 能力：在用户**自行设定的时段**，提醒其完成当日尚未完成的打卡（饮食 / 运动 / 用药 / 血糖），并与个人中心「隐私与通知 → **打卡提醒**」开关（`privacy_settings.checkin_notify`）联动。

### 1.2 当前实现（截至 v1.0 文档）

| 能力 | 状态 |
|------|------|
| 个人中心「打卡提醒」开关 | ✅ 已实现（`checkin_notify` 写入 `USERS.PRIVACY_SETTINGS`） |
| 首页「打卡提醒」快捷入口 | ✅ 已实现（跳转个人中心并定位到开关） |
| 用户自定义提醒时段 | ❌ 待实现 |
| 到点提醒触达 | ❌ 待实现 |
| 提醒规则持久化 | ❌ 待实现 |

### 1.3 设计目标

| 目标 | 说明 |
|------|------|
| 按时提醒 | 在用户设定的时段检查当日对应类型是否已打卡，未完成则提醒 |
| 用户可控 | 全局开关 + 分类型/分时段规则，均可关闭或调整 |
| 低打扰 | 同一时段、同一类型每日最多提醒 1 次（可配置稍后提醒，见 §5） |
| 可闭环 | 提醒点击后直达生活打卡页对应 Tab，减少操作路径 |

### 1.4 与健康打卡模块的边界

| 能力 | 打卡提醒模块 | 生活打卡（CheckinRecords） |
|------|--------------|----------------------------|
| 设定提醒时间 | ✅ | ❌ |
| 到点通知 | ✅ | ❌ |
| 录入打卡数据 | ❌ | ✅ |
| 积分 / 成就 | 不写入 | ✅ 沿用现有逻辑 |
| 用药打卡 Tab | 仅跳转引导 | ✅ 选用/自定义用药记录 |

[健康打卡模块产品设计说明书.md](./健康打卡模块产品设计说明书.md) 专注打卡录入与展示；**打卡提醒、个人中心通知开关联动**由本文档定义。

## 2. 提醒对象与触发条件

### 2.1 可提醒的打卡类型

与现有 `CHECKIN_RECORDS.CHECKIN_TYPE` 对齐：

| 类型值 | 名称 | 典型提醒场景 |
|--------|------|--------------|
| 1 | 饮食打卡 | 早/午/晚/加餐时段后尚未记录 |
| 2 | 运动打卡 | 用户设定的运动计划时间后尚未记录 |
| 3 | 用药打卡 | 用户设定的服药时间后尚未记录 |
| 4 | 血糖打卡 | 空腹/餐后等监测时间点尚未记录 |

### 2.2 触发逻辑（核心规则）

在规则配置的 **`remind_time`（HH:mm）** 到达时，系统执行：

```
IF 用户 checkin_notify == false → 不提醒
ELSE IF 当日该 checkin_type 已有至少 1 条记录 → 不提醒
ELSE IF 当日该规则已触发过且用户未点「稍后提醒」→ 不重复提醒
ELSE → 按 §5 选定的方式发送提醒
```

说明：

- **「当日已打卡」** 以 `CHECKIN_RECORDS` 中 `USER_ID + CHECKIN_TYPE + CHECKIN_DATE = 今天` 是否存在记录为准（与打卡模块「同日可多条饮食记录」策略一致：只要该类型当日有记录即视为已完成，不再提醒）。
- **跨日重置**：每日 00:00（用户时区，默认 `Asia/Shanghai`）清空「已触发」标记。

### 2.3 默认推荐时段（首次开启引导）

用户首次打开「打卡提醒」或规则为空时，可一键应用推荐模板（可修改）：

| 打卡类型 | 默认时段 | 说明 |
|----------|----------|------|
| 饮食 | 08:00、12:00、18:00 | 对应早/午/晚 |
| 用药 | 08:00、20:00 | 常见早晚服药 |
| 运动 | 17:30 | 下班后 |
| 血糖 | 07:00、10:00 | 空腹、早餐后 |

## 3. 提醒方式（重点）

系统采用 **分层触达**：优先使用对用户干扰小、实现成本低的通道；仅在用户授权且开关打开时使用系统级通知。

### 3.1 方式总览

| 优先级 | 方式 | 代号 | 适用场景 | 依赖 |
|--------|------|------|----------|------|
| P0 | **应用内消息** | `in_app` | 用户正在浏览本站任意页面 | 前端轮询或 WebSocket |
| P0 | **打卡页内横幅** | `in_page_banner` | 用户已在 `/checkin-records` | 前端本地调度 |
| P1 | **浏览器系统通知** | `browser_push` | 用户已授权 Notification 且标签页在后台 | Notification API + 用户授权 |
| P2 | **稍后提醒** | `snooze` | 用户点击「15 分钟后提醒」 | 前端/后端 snooze 队列 |

> **原则**：`checkin_notify = false` 时，**所有方式均不触发**；规则配置页仍可编辑，但不发送任何提醒。

### 3.2 P0 — 应用内消息（`in_app`）

**形态**：Element Plus `ElNotification`（右上角），停留 8 秒，可手动关闭。

**文案示例**：

| 类型 | 标题 | 正文 |
|------|------|------|
| 饮食 | 饮食打卡提醒 | 尚未记录今日饮食，点击去完成打卡 |
| 用药 | 用药打卡提醒 | 尚未记录今日用药，点击去完成打卡 |
| 运动 | 运动打卡提醒 | 尚未记录今日运动，点击去完成打卡 |
| 血糖 | 血糖打卡提醒 | 尚未记录今日血糖，点击去完成打卡 |

**交互**：

- 主按钮「去打卡」→ 跳转 `/checkin-records?tab={diet|medication|exercise|glucose}`
- 次按钮「稍后提醒」→ 见 §3.5

**技术要点**：

- 登录状态下，前端每 **60 秒** 调用 `GET /api/v1/checkin/reminders/pending` 获取当前应展示的提醒（服务端根据服务器时间与用户规则计算，避免纯客户端时钟漂移）。
- 同一 `reminder_log_id` 仅展示一次，关闭后调用 `POST .../ack` 标记已读。

### 3.3 P0 — 打卡页内横幅（`in_page_banner`）

**形态**：`CheckinRecords` 页面顶部固定横幅（`el-alert` type="warning"），列出当前待完成类型。

**适用**：用户已打开打卡页时，**不再弹出** `ElNotification`，改为横幅，避免重复打扰。

**交互**：横幅内「立即打卡」切换到对应 Tab。

### 3.4 P1 — 浏览器系统通知（`browser_push`）

**前置条件**（全部满足才发送）：

1. `checkin_notify === true`
2. 用户曾点击「开启浏览器通知」且 `Notification.permission === 'granted'`
3. 文档处于后台或最小化（`document.visibilityState === 'hidden'`）**或** 用户勾选「即使在前台也使用系统通知」

**形态**：原生 `new Notification(title, { body, icon, tag })`

| 字段 | 值 |
|------|-----|
| `tag` | `checkin-reminder-{type}-{date}` | 同 tag 覆盖，避免堆叠 |
| `icon` | 站点 favicon |
| 点击 | `window.focus()` + 跳转打卡页对应 Tab |

**权限引导**：个人中心「打卡提醒」开关旁增加「开启浏览器通知」链接；首次开启打卡提醒时可弹窗引导（可跳过）。

### 3.5 P2 — 稍后提醒（`snooze`）

用户点击「稍后提醒」后：

- 默认 **15 分钟** 后再次检查；若仍未打卡且 `checkin_notify` 仍为 true，则按 §3.2 / §3.4 再次触达。
- 同一规则、同一自然日 **最多 snooze 2 次**，之后仅保留打卡页横幅（若用户访问打卡页）。

### 3.6 触达方式决策流程

```mermaid
flowchart TD
    A[到达 remind_time] --> B{checkin_notify?}
    B -->|否| Z[结束]
    B -->|是| C{当日该类型已打卡?}
    C -->|是| Z
    C -->|否| D{今日已提醒且未 snooze?}
    D -->|是| Z
    D -->|否| E{用户在 /checkin-records?}
    E -->|是| F[in_page_banner]
    E -->|否| G{visibility hidden 且浏览器通知已授权?}
    G -->|是| H[browser_push]
    G -->|否| I[in_app ElNotification]
    F --> J[用户点击去打卡]
    H --> J
    I --> J
    J --> K[/checkin-records?tab=...]
```

## 5. 数据模型（规划）

库：`DIABETES_CHECKIN`

### 5.1 提醒规则表 `CHECKIN_REMINDER_RULES`

```sql
CREATE TABLE CHECKIN_REMINDER_RULES (
    RULE_ID        VARCHAR(32)  NOT NULL PRIMARY KEY,
    USER_ID        VARCHAR(32)  NOT NULL,
    CHECKIN_TYPE   TINYINT      NOT NULL COMMENT '1饮食 2运动 3用药 4血糖',
    REMIND_TIME    TIME         NOT NULL COMMENT '每日提醒时刻 HH:mm:ss',
    ENABLED        TINYINT(1)   NOT NULL DEFAULT 1,
    SORT_ORDER     INT          NOT NULL DEFAULT 0,
    CREATED_AT     DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UPDATED_AT     DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_user_type (USER_ID, CHECKIN_TYPE)
) COMMENT='打卡提醒规则';
```

### 5.2 提醒触达日志 `CHECKIN_REMINDER_LOGS`（可选，便于去重与统计）

```sql
CREATE TABLE CHECKIN_REMINDER_LOGS (
    LOG_ID         VARCHAR(32)  NOT NULL PRIMARY KEY,
    USER_ID        VARCHAR(32)  NOT NULL,
    RULE_ID        VARCHAR(32)  NOT NULL,
    CHECKIN_TYPE   TINYINT      NOT NULL,
    REMIND_DATE    DATE         NOT NULL,
    CHANNEL        VARCHAR(32)  NOT NULL COMMENT 'in_app/browser_push/in_page_banner',
    STATUS         TINYINT      NOT NULL DEFAULT 0 COMMENT '0已发送 1已点击 2已忽略 3snooze',
    SNOOZE_UNTIL   DATETIME     DEFAULT NULL,
    CREATED_AT     DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_rule_date (RULE_ID, REMIND_DATE)
) COMMENT='打卡提醒触达日志';
```

## 6. API 设计（规划）

**服务**：`checkin-service`  
**前缀**：`/api/v1/checkin/reminders`  
**鉴权**：JWT

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/rules` | 获取当前用户全部提醒规则 |
| PUT | `/rules` | 批量保存规则（覆盖某类型或全量） |
| GET | `/pending` | 返回当前时刻应展示的待提醒列表（供前端轮询） |
| POST | `/logs/{logId}/ack` | 标记已读/已忽略 |
| POST | `/logs/{logId}/snooze` | 稍后提醒，body: `{ "minutes": 15 }` |
| GET | `/defaults` | 返回 §2.3 推荐时段模板 |

**与 user-service 协作**：

- `GET /pending` 内部通过 `UserServiceClient` 读取 `privacy_settings.checkin_notify`；为 false 时返回空数组。

## 11. 与旧「用药提醒」方案差异

| 维度 | 旧方案（已废弃） | 本方案 |
|------|------------------|--------|
| 提醒对象 | 仅服药 | 饮食 / 运动 / 用药 / 血糖四类打卡 |
| 核心动作 | 确认已服 → 自动写入用药打卡 | 提醒 → 用户自行进入打卡页录入 |
| 独立路由 | `/medication-reminders` | `/checkin-reminder-settings` |
| 通知开关 | 与打卡提醒混用 | 统一使用 `checkin_notify` |
| 数据表 | `MEDICATION_REMINDERS*` | `CHECKIN_REMINDER_RULES*` |
