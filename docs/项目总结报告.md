# 糖尿病智能助手 — 项目总结报告

## 1. 项目概述

### 1.1 项目背景

糖尿病已成为全球性公共卫生挑战。世界卫生组织数据显示，全球糖尿病患者数量已达数亿并持续攀升。在此背景下，项目依托国家"健康中国 2030"慢性病管理政策导向，创新性地融合 **Dify 智能体编排平台**、**DeepSeek 大模型**、**Milvus 向量数据库** 等前沿 AI 技术，构建了一套面向糖尿病患者及高危人群的预治一体化智能管理平台。

### 1.2 系统定位

本系统定位为**基于大模型和智能体技术的糖尿病预治一体化管理平台**，提供从"科普教育 → 风险评估 → AI 问诊 → 方案生成 → 日常打卡 → 行为分析 → 资讯推荐 → 主动干预"的完整健康管理闭环。核心价值主张：

- **智能驱动**：以 DeepSeek 大模型为核心推理引擎，实现风险预测、方案生成、AI 问诊等复杂能力；
- **知识加持**：以 Milvus 医学知识库（DiaKG 2292 段、28 万字）为 RAG 数据底座，确保 AI 输出专业合规；
- **全链路覆盖**：8 大核心功能模块 + 6 大 AI 扩展功能，覆盖预防、诊断辅助、管理、干预全周期；
- **私有化部署**：支持 Docker Compose 一键部署，医疗数据不出私有网络。

## 4. AI 与大模型技术应用

### 4.1 工作流总览

系统通过 Dify 平台编排 **10+ 个工作流**，按业务场景分为三类调用模式：

| 序号 | 工作流 | 调用方 | 模式 | RAG | 优先级 |
|------|--------|--------|------|-----|--------|
| 1 | 打卡 AI 行为分析 | checkin-service | blocking | 否 | P0 ✅ |
| 2 | 健康方案生成 | plan-service | SSE 流式（后端拆分） | 否 | P0 ✅ |
| 3 | 糖尿病风险评估 | health-service | blocking | 否 | P0 ✅ |
| 4 | 资讯个性化推荐 | article-service | blocking | 是（Milvus） | P1 ✅ |
| 5 | 科普问答（Chatbot） | home-service | SSE 流式 | 是（Milvus） | P0 ✅ |
| 6 | AI 医生问诊 | consultation-service | blocking | 是（Milvus） | P0 ✅ |
| 7 | 健康趋势分析 | user-service | blocking | 否 | P1 设计完成 |
| 8 | 语音识别（STT） | home-service | blocking | 否 | P1 设计完成 |
| 9 | 资讯初稿生成 | admin-frontend 代理 | SSE 流式 | 是（Milvus） | P1 ✅ |
| 10 | 食物图片识别 | checkin-service | blocking | 否 | P2 设计完成 |
| 11 | 健康画像 Agent | user-service | blocking（Agent 多工具） | 是（Milvus） | P2 设计完成 |

**契约驱动模式**：每个工作流对应 Java Contract 类（`*WorkflowContract.java`）和 JSON Schema 文件（`*.schema.json`），确保 AI 层与业务层数据格式一致。提供统一的 `dify-workflow-spec` 查询 API 便于联调。

### 4.2 RAG 检索增强生成

- **医学知识库**：基于 DiaKG 41 篇诊疗指南 JSON（2292 段、28 万字），经纯文本提取、分段（400~600 字/chunk，80 字 overlap）、Embedding 向量化后写入 Milvus `diabetes_knowledge` Collection（1024 维，COSINE 距离度量）
- **资讯语义库**：`article_knowledge` Collection（384 维），用于资讯语义相似检索
- **检索流程**：用户查询 → Embedding 向量化 → Milvus ANN 检索 Top-K → 相似度过滤（≥0.7） → 拼接 `knowledge_context` → 注入 Dify 工作流
- **降级策略**：Milvus 不可用或无结果时，`knowledge_context=""`，LLM 基于通用知识回答并标注来源

### 4.3 创新业务闭环

**1. 主动健康干预闭环**（Phase 1 设计完成）：

```
被动记录（健康档案/打卡）→ 智能检测（规则引擎 + Dify 趋势分析）
→ 干预编排（消息预警 + 提醒策略调整）→ 主动触达（消息中心 health_alert + 动态打卡提醒）
→ 行为反馈（用户查看预警 → 打卡/更新档案 → 回到起点）
```

- **双轨决策**：本地医学阈值规则（如空腹血糖 ≥ 7.0 → warning）+ Dify 趋势 AI，取较高严重等级，LLM 不可降级 critical 级别
- **低打扰设计**：Redis 冷却期 4h、每日 health_alert 上限 2 条（critical 除外）、24h 同类型 upsert

**2. 用户健康画像 Agent**（Phase 2 设计完成）：

- 聚合风险、打卡、方案、知识四域数据为统一"数字健康孪生体"
- Dify Agent 按需调用 4 个 HTTP 工具（`get_risk_history`、`get_checkin_trends`、`get_plan_compliance`、`search_knowledge`）
- 每条洞察附带 `evidence`（数据来源 + 指标快照），支持可解释性

## 7. 创新亮点

1. **契约驱动的 AI 工程化**：为每个 Dify 工作流建立 Java Contract 类 + JSON Schema + 契约查询 API，确保 AI 层与业务层数据格式严格一致，降低集成风险。

2. **Milvus 医学知识库 RAG 体系**：41 篇 DiaKG 诊疗指南经 ETL 纯文本化 → 分段 → Embedding → Milvus 入库，为科普问答、AI 问诊、方案生成等 5+ 场景提供权威医学知识增强。

3. **主动健康干预闭环**：将规则引擎与 Dify 趋势分析双轨合并，实现"被动记录 → 智能检测 → 干预编排 → 主动触达 → 行为反馈"的完整闭环，并设计 Redis 去重冷却、每日上限等防打扰机制。

4. **用户健康画像 Agent**：跨风险、打卡、方案、知识四域数据聚合，通过 Dify Agent 多工具编排生成可解释画像，每条洞察附带 `evidence` 溯源。

5. **SSE 分阶段流式方案生成**：后端以 blocking 调用 Dify，收到完整方案后拆分 4 个 SSE 事件（热量 → 饮食 → 运动 → 作息）推送给前端，兼顾 Dify 集成简洁性与前端交互体验。

6. **多层次优雅降级**：Dify 不可用时自动切换本地规则引擎/模板回复；Milvus 不可用时使用 LLM 通用知识；路径覆盖全部 AI 入口。

7. **测试驱动 + 覆盖率门禁**：后端 JaCoCo + 前端 Vitest 覆盖率门禁强制执行，全量 1212 + 86 条用例，核心业务包 100% 行 + 分支覆盖。

8. **国产化适配**：审计日志独立使用金仓 KingbaseES 数据库，与业务 MySQL 物理隔离，满足合规与国产化要求。
