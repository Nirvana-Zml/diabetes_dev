| 2    | 医生在线咨询       | 连接专业内分泌科医生资源，提供实时一对一在线问诊服务，支持图文、语音沟通，快速解答用户疑虑，建立高效医患沟通桥梁。支持医生列表浏览、科室筛选、即时咨询/留言、会话评价等功能。 |
| 3    | 糖尿病风险预测     | 基于大数据与深度学习构建糖尿病风险评估模型，通过用户录入身体指标（空腹血糖、BMI、血压等）与生活习惯（饮食、运动、吸烟饮酒等），智能计算患病概率，生成可视化风险报告（含风险等级、因素分析、改善建议）。 |

| 5    | 生活打卡功能       | 设置饮食、运动、用药、血糖监测四大打卡维度，支持一键记录每日健康行为。通过积分奖励与连续打卡激励机制（连续≥7天发放成就徽章），帮助用户建立良好的健康习惯。 |
| 6    | 个人中心管理       | 一站式整合用户基础信息、健康档案（身高/体重/BMI/血糖/血压/病史）、就诊记录与方案数据，支持账户信息修改、头像上传、隐私设置与数据导出（PDF/Excel），保障用户数据安全与管理便捷性。 |
| 7    | 打卡信息管理       | 自动统计用户历史打卡数据，生成周/月度行为趋势图表（饮食/运动/用药/血糖四项指标），直观展示健康行为完成度。支持数据回溯与异常提醒，辅助用户分析行为规律。 |
| 8    | 健康资讯管理       | 构建专业内容审核与发布体系，智能推送精准、及时的健康资讯与疾病护理知识，支持按用户标签个性化分发，提升内容触达效率。包含资讯列表、标签筛选/搜索、详情阅读等功能。 |

### b. 性能

| 性能指标             | 要求说明                                                                                                                                     |
| -------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| **响应时间**         | 普通页面加载时间 ≤ 2秒；AI问答首Token响应时间 ≤ 3秒（SSE流式传输）；风险预测报告生成时间 ≤ 15秒；数据导出异步生成，完成后通知用户下载。       |
| **并发处理能力**     | 系统需支持至少100个并发用户的正常访问；Dify智能体层需支持至少20个并发AI推理请求；MySQL数据库连接池支持至少50个并发连接。                        |
| **存储容量**         | 业务数据库（MySQL）：初期规划100GB，支持在线扩容至500GB；向量知识库（Milvus）：初期规划200GB，用于存储医学文献、诊疗指南的向量数据；对象存储（MinIO）：初期规划100GB，用于用户上传的图片、报告等文件。 |
| **吞吐量**           | API网关（Nginx）需支持至少500 requests/second的吞吐能力；Dify工作流处理能力 ≥ 30 workflows/minute。                                            |
| **可用性**           | 系统目标可用性 ≥ 99.5%（月故障时间 ≤ 3.6小时）；核心AI推理链路支持失败自动重试（最多3次）；知识库检索支持降级策略（无结果时使用通用知识）。       |
| **资源占用**         | 服务端总计最低配置：24 Core CPU、46 GB 内存、510 GB 存储；客户端浏览器最低支持 Chrome 90+ / Edge 90+ / Safari 15+，屏幕分辨率 ≥ 1366×768。       |
| **缓存策略**         | 使用Redis 7.2缓存热点数据（如科普首页内容、医生列表），缓存过期时间根据数据类型设置为5~30分钟，降低数据库查询压力。                                |
| **流式传输**         | AI回答采用SSE（Server-Sent Events）流式传输，实现逐Token实时渲染，避免用户长时间等待。                                                           |

### e. 基本的数据流程和处理流程

**整体数据流走向（四层架构）：**

```
用户浏览器（Vue 3 前端）
    ↕ HTTPS REST API / SSE
Dify 智能体层（意图识别 → 工作流编排 → 对话管理 → 工具调度）
    ↕ API调用                          ↕ 向量检索API          ↕ 业务数据读写
DeepSeek大模型层                    Milvus知识库层          MySQL + Redis
(DeepSeek-V3通用推理/               (医学文献/诊疗指南/    (业务数据/缓存)
 DeepSeek-R1深度推理)                 药典向量数据)
```

**核心处理流程：**

1. **科普问答流程**：用户输入问题 → Dify意图识别（科普类）→ Milvus语义检索（Top-5文档片段）→ 组装Prompt → DeepSeek-V3生成回答 → SSE流式返回前端 → Markdown渲染（含引用来源）→ 安全过滤后展示。

2. **在线问诊流程**：用户选择医生发起咨询 → 建立会话 → 用户发送消息（图文/语音）→ Dify问诊工作流解析症状 → 检索知识库诊疗指南 → DeepSeek-V3生成AI参考建议（仅医生端可见）→ 医生确认/修改后发送回复 → 用户收到回复 → 结束咨询并评价。

3. **风险评估流程**：用户填写健康指标表单 → 必填项校验 → Dify风险评估工作流 → 调用医学指标计算器工具（BMI/血糖等级/基础风险分）→ 组装Prompt → DeepSeek-R1深度推理 → 结构化JSON评估结果 → 生成可视化报告（ECharts）→ 展示风险等级 + 因素分析 + 改善建议 → 保存评估记录。

4. **方案生成流程**：收集用户健康数据（健康档案 + 风险评估 + 打卡数据）→ Dify方案生成工作流 → 检索用户画像标签 → 检索知识库饮食/运动指南 → 调用每日热量计算器工具（Mifflin-St Jeor公式）→ DeepSeek-V3生成四大维度方案（饮食/运动/作息/用药）→ 方案结构化拆分 → 组装方案卡片展示。

5. **生活打卡流程**：用户选择打卡类型（饮食/运动/用药/血糖）→ 填写打卡详情 → 提交打卡记录 → 积分计算 → 连续打卡检测（≥7天发放徽章）→ 更新积分与成就 → 前端更新打卡状态 + 鼓励动画。

6. **资讯推送流程**：后台编辑发布资讯 → 内容审核 → 用户标签匹配 → 个性化分发 → 用户浏览资讯列表 → 标签筛选/搜索 → 阅读详情。

**数据存储流程：**
- 业务数据（用户信息、健康档案、打卡记录、问诊记录、咨询会话、方案数据、资讯内容）→ MySQL 8.0 关系型数据库
- 热点数据（首页内容、医生列表、会话状态）→ Redis 7.2 缓存
- 医学知识（病例、药典、文献、诊疗指南）→ 分段 → BAAI/bge-large-zh-v1.5 Embedding → Milvus 2.4 向量数据库
- 用户上传文件（头像、检查报告、咨询图片）→ MinIO 对象存储

### f. 安全与保密要求

| 序号 | 安全维度               | 具体要求                                                                                                                                     |
| ---- | ---------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| 1    | **数据传输安全**       | 全链路采用HTTPS/TLS 1.3加密传输；前端与Dify之间使用HTTPS REST API通信；敏感数据禁止明文传输。                                                  |
| 2    | **数据存储安全**       | 用户健康数据、问诊记录、个人身份信息采用AES-256加密存储；遵循《个人信息保护法》最小必要原则，仅收集实现功能所必需的用户数据。                      |
| 3    | **身份认证**           | 采用JWT（JSON Web Token）令牌机制进行用户身份认证；支持令牌过期自动刷新；敏感操作（如数据导出、密码修改）需二次身份验证。                        |
| 4    | **权限分级控制**       | 基于RBAC（基于角色的访问控制）实现三级权限体系：普通用户（查看自己的数据、使用前端功能）、医生（查看患者数据、回复咨询）、管理员（系统配置、内容审核、用户管理）。 |
| 5    | **AI内容安全**         | AI生成内容需经过安全过滤：自动检测并拦截违规内容（如违法用药建议、虚假医疗信息）；所有AI回答附带"仅供参考，不构成诊疗建议"免责声明；超出AI能力范围的问题（如开具处方）自动引导转在线问诊。 |
| 6    | **接口安全**           | API接口实施防刷机制（频率限制、验证码）；请求参数前后端双重校验，防止SQL注入、XSS攻击；接口调用日志全量记录，支持操作留痕和审计追溯。              |
| 7    | **隐私保护**           | 用户可自主设置个人数据可见性；支持数据导出（用户数据可携带权）和账户注销（被遗忘权）；医生仅能查看其接诊患者的相关数据，不可跨患者访问。            |
| 8    | **法律法规遵从**       | 遵循《个人信息保护法》（2021）、《数据安全法》（2021）、《健康中国2030规划纲要》等法律法规；AI医疗建议遵循《中国2型糖尿病防治指南》（2024版）等权威规范。 |
| 9    | **私有化部署安全**     | 系统支持本地/私有云部署，医疗数据不出私有网络环境；Dify和DeepSeek支持私有化部署与API直连，兼顾数据安全与开发效率。                               |
| 10   | **日志与审计**         | 所有数据操作（增删改查）记录操作留痕日志，包括操作人、操作时间、操作类型、操作内容；日志定期归档，支持安全审计与异常行为回溯。                     |

## 2.4 可行性研究方法

### 2.4.1 研究方法

| 方法类型         | 具体说明                                                                                                                     |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| **需求调查法**   | 通过分析项目背景文档、设计概述、系统详细设计说明书等内部文档，全面梳理系统的功能需求、性能需求、安全需求，确定项目的需求边界和优先级。 |
| **技术调研法**   | 对项目拟采用的核心技术栈（DeepSeek大模型、Dify智能体平台、Milvus向量数据库、Vue 3前端框架等）进行技术成熟度、社区活跃度、文档完备性、学习成本等方面的调研评估。 |
| **对比分析法**   | 将本系统方案与市场上已有的糖尿病管理类产品进行功能、技术、成本、用户体验等多维度对比，确立本系统的差异化竞争优势。                   |
| **加权评分法**   | 建立多维度评价指标体系（功能完整性、技术可行性、经济可行性、法律合规性、操作可行性），对各维度进行加权评分，综合判定项目的总体可行性。 |
| **成本估算法**   | 基于系统架构和服务端环境要求，估算硬件资源、云服务、开发人力、数据采购等各项投入成本，结合经费来源评估经济可行性。               |
| **风险评估法**   | 识别项目开发过程中可能面临的技术风险（如大模型调用稳定性、知识库质量）、进度风险（如开发周期紧张）、合规风险（如医疗法规遵从），并制定相应的风险应对策略。 |
| **原型验证法**   | 对核心AI功能链路（科普问答、风险评估、方案生成）搭建最小可行原型（MVP），通过实际运行验证技术方案的可行性和响应性能。             |

| **技术可行性**     | 评估各项技术选型的成熟度、兼容性、性能表现、团队技术储备                                                                      | 所有技术组件均有成功案例支撑，团队具备实施能力   |
| **经济可行性**     | 评估开发成本、部署成本、运营成本与预期收益（教学价值、社会价值）的对比                                                        | 总成本在经费预算范围内，投入产出比合理           |
| **法律合规性**     | 评估系统在个人信息保护、医疗合规、数据安全等方面的法律合规程度                                                                | 符合《个人信息保护法》《数据安全法》等法律法规   |
| **操作可行性**     | 评估目标用户（中老年糖尿病患者）对系统的接受度、学习成本和使用便利性                                                          | 核心功能操作路径 ≤ 3步，界面适配中老年用户习惯   |
| **进度可行性**     | 评估项目在给定时间约束内完成开发和部署的可能性                                                                                | 开发周期 ≤ 30周，可按阶段分期交付               |
| **维护可行性**     | 评估系统上线后的运维复杂度、技术债务风险和版本迭代能力                                                                        | 模块化架构支持独立升级，运维文档完备              |

### 2.4.3 基本策略

- **建立基准点**：以"糖尿病风险预测准确率 ≥ 85%"、"AI科普回答引用来源准确率 ≥ 95%"、"用户满意度 ≥ 4.0/5.0"等作为系统成功的核心基准点。
- **仿真验证**：通过模拟 100 并发用户的压力测试，验证系统在真实负载下的性能表现是否满足SLA指标。
- **迭代评估**：在每个开发阶段结束时进行阶段性可行性复审，根据实际进展和技术验证结果动态调整后续计划。

## 2.5 决定可行性的主要因素

对系统进行评价时，将采用以下主要尺度进行综合衡量：

| 序号 | 评价尺度             | 权重     | 说明                                                                                                                                   |
| ---- | -------------------- | -------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| 1    | **费用的多少**       | **25%**  | 项目的总拥有成本（TCO），包括开发人力成本、云服务器资源费用、DeepSeek API调用费用（当前免费）、医学知识库采购费用、第三方服务费用等。经济可行性是项目能否启动的首要考量因素。 |
| 2    | **各项功能的优先次序** | **25%**  | 核心功能（科普首页、在线咨询、风险预测、方案生成、打卡、个人中心、打卡管理、资讯管理）的优先级排序和实现程度。优先确保高频刚需功能（风险预测、方案生成、在线咨询）的完整实现，次要功能可在后续版本迭代中完善。 |
| 3    | **开发时间的长短**   | **20%**  | 项目从立项到投入使用的总周期（预计30周），以及各里程碑节点的按时达成率。时间可行性直接影响项目能否在学期教学周期内完成并交付使用。         |
| 4    | **使用中的难易程度** | **15%**  | 目标用户（包括中老年糖尿病患者、医生、管理员）对系统的操作便利性和学习成本。系统需提供简洁直观的UI界面、清晰的操作引导、适配中老年用户的大字号和高对比度设计。 |
| 5    | **技术实现的成熟度** | **10%**  | 核心AI功能（DeepSeek大模型调用、Dify工作流编排、RAG知识库检索、SSE流式响应）的技术可行性和稳定性。DeepSeek和Dify均为成熟产品，有丰富的文档和社区支持，技术风险较低。 |
| 6    | **数据安全与合规性** | **5%**   | 系统在医疗数据隐私保护、AI内容安全、法律法规遵从等方面的合规程度。涉及用户健康敏感数据的系统，安全合规是红线要求。                       |

**综合评价结论：**

基于以上六大评价尺度的综合分析，本项目在费用方面因采用DeepSeek免费商用和全开源技术栈而具有显著的经济优势；功能优先级清晰，核心功能技术方案明确；30周开发周期充裕可控；系统针对中老年用户进行了易用性优化设计；核心技术组件成熟可靠；数据安全设计了完整的防护体系。**综合评估认为，本项目具备较高的可行性，建议批准立项并按计划推进。**
