# 大模型技术在糖尿病智能助手系统中的使用说明

## 1. 概述

本文档说明在糖尿病智能助手系统中使用大模型（LLM）技术的设计方案、调用方式与开发规范。系统以 **Dify** 作为大模型应用编排平台，后端通过统一 `DifyClient` 进行工作流调用，前端在特定场景可通过后端提供的配置直调 Dify。

### 1.1 大模型定位

| 层次 | 技术/平台 | 用途 |
|------|-----------|------|
| 推理模型 | DeepSeek-V4 | 深度推理、文本生成、逻辑分析 |
| 编排平台 | Dify（自部署实例） | 工作流编排、Prompt 管理、知识库检索 |
| 向量检索 | Milvus + BGE Embedding | 医学知识库语义检索 |
| 后端集成 | `DifyClient`（`backend/common`） | 统一 HTTP 调用封装 |
| 前端直调 | 通过后端配置接口获取 Dify URL | 资讯 AI 辅助生成（SSE 流式） |

## 3. 核心架构组件

### 3.1 DifyClient —— 统一 HTTP 调用客户端

`DifyClient` 位于 `backend/common` 模块，所有后端服务通过它调用 Dify 工作流。

**核心职责：**
- 统一封装 `POST /v1/workflows/run` 请求
- 处理 `blocking` / `streaming` 两种响应模式
- 注入 `Authorization: Bearer {API_KEY}` 认证头
- 管理超时、重试、错误处理

**使用示例（Java）：**

```java
// 1. 注入 DifyClient
@Autowired
private DifyClient difyClient;

// 2. 构建请求 payload
Map<String, Object> inputs = new HashMap<>();
inputs.put("query", "请分析用户的打卡行为数据");
inputs.put("user_id", userId);
inputs.put("start_date", "2024-06-01");
// ...

// 3. 调用工作流
DifyResponse response = difyClient.runWorkflow(
    apiKey,          // 从环境变量获取
    inputs,          // 业务 payload
    userId,          // 用户标识
    "blocking"       // 响应模式
);

// 4. 解析输出
JsonNode outputs = response.getOutputs();
String analysis = outputs.get("behavior_analysis").get("summary").asText();
```

### 3.2 DifyJsonSchema —— Schema 构建工具

`DifyJsonSchema` 位于 `backend/common`，用于构建符合 Dify 规范的 JSON Schema（用于工作流开始节点配置）。

**核心约定：**
- 顶层及所有嵌套 `object` 必须包含 `"required": []` 与 `"additionalProperties": true`
- 字符串字段仅 `{ "type": "string" }`
- 数值字段 `{ "type": "number" }` 或 `{ "type": "integer" }`

### 3.3 Milvus 向量知识库

系统使用 **Milvus** 存储医学知识库的向量索引，通过 **BGE Embedding** 模型将文档片段向量化。

**检索流程：**
```
用户查询 → BGE Embedding 向量化 → Milvus ANN 检索 Top-K → 返回文档片段
```

## 4. 工作流调用模式

### 4.1 模式一：阻塞模式（Blocking）

**适用场景：** 期望一次返回完整结果的场景（风险评估、行为分析、AI 问诊等）。

**请求：**
```json
{
  "response_mode": "blocking",
  "user": "usr_xxx",
  "inputs": { ... }
}
```

**响应：**

```json
{
  "data": {
    "status": "succeeded",
    "outputs": {
      "<output_key>": { ... }
    }
  }
}
```

**后端解析顺序：** `data.outputs.<key>` → `outputs.<key>` → `outputs.text` 内嵌 JSON。

### 4.2 模式二：流式模式（Streaming / SSE）

**适用场景：** 需要逐步返回内容的场景（科普问答、方案生成、资讯初稿）。

**请求：** `"response_mode": "streaming"`

**响应（SSE 事件流）：**
```
event: message
data: {"event": "message", "answer": "这是流式返回的内容片段..."}

event: message_end
data: {"event": "message_end", "metadata": {...}}
```

**后端处理方式：**

- **方案生成（plan-service）：** 后端以 blocking 模式调用 Dify，收到完整结果后，由后端拆分为多个 SSE 事件（`stage_calorie` → `stage_diet` → `stage_exercise` → `stage_rest` → `complete`）向前端推送。
- **科普问答（home-service）：** 后端透传 Dify SSE 流，前端直接渲染 Markdown。

### 4.3 模式三：前端直调（资讯初稿生成）

**适用场景：** 管理端需要实时流式展示 AI 生成内容的场景。

**流程：**
1. 前端调用 `GET /api/admin/articles/ai-draft/config` 获取 Dify 工作流 URL + API Key
2. 前端直接以 SSE 方式调用 Dify 工作流
3. Dify 流式返回生成内容，前端实时渲染可编辑区域

## 5. 入参传递方式

### 5.1 双层嵌套（`inputs.inputs`）

适用于开始节点使用单个 `inputs` Object 变量（对象类型）的工作流。

```json
{
  "inputs": {
    "inputs": {
      "query": "...",
      "user_id": "..."
    }
  }
}
```

### 5.2 平铺模式（Flat）

适用于开始节点使用多个独立变量的工作流。各字段直接写入 `inputs`。

```json
{
  "inputs": {
    "query": "...",
    "user_id": "...",
    "checkin_stats": { ... }
  }
}
```

**使用平铺模式的工作流：**
- 打卡 AI 行为分析
- 健康方案生成
- 糖尿病风险评估（`DIFY_RISK_INPUT_VAR=flat`）
- 资讯个性化推荐（`DIFY_ARTICLE_RECOMMEND_INPUT_VAR=flat`）

### 5.3 JSON 字符串传递

部分工作流（如风险评估、资讯推荐）中，复杂对象字段作为 **JSON 文本字符串** 传入，在 Dify 内部通过 Code 节点 `JSON.parse` 解析。

```json
{
  "inputs": {
    "user_profile": "{\"age\":48,\"gender\":\"male\"}",
    "questionnaire": "{\"height\":170,\"weight\":82,...}"
  }
}
```

## 6. 各模块大模型集成详解

### 6.1 科普展示首页（home-service）

| 大模型能力 | 说明 |
|-----------|------|
| **AI 科普问答** | 调用科普问答工作流，SSE 流式返回 Markdown 回答 |
| **知识库检索** | Milvus 向量检索糖尿病科普知识库 Top-5 文档片段 |
| **个性化推荐** | 基于用户偏好标签的内容推荐 |

**交互流程：** 用户提问 → BGE Embedding 向量化 → Milvus ANN 检索 → Dify 组装 Prompt（系统指令 + 知识上下文 + 用户问题）→ DeepSeek-V4 推理 → SSE 流式返回。

### 6.2 医生在线咨询（consultation-service）

| 大模型能力 | 说明 |
|-----------|------|
| **AI 医生模拟** | 基于预设角色设定，模拟医生问诊 |
| **诊疗知识检索** | 检索糖尿病诊疗指南、药物信息 |
| **结构化回复** | 返回可能诊断、追问问题、建议检查、治疗策略 |

**关键输入：**

- `doctor_role`：AI 医生角色设定（如"资深内分泌科主任医师"）
- `patient_profile`：患者基本信息与健康档案
- `conversation_history`：当前会话历史（最近 20 条）
- `knowledge_context`：知识库检索的诊疗指南片段

### 6.3 糖尿病风险评估（health-service）

| 大模型能力 | 说明 |
|-----------|------|
| **深度风险推理** | 基于医学指标与风险因素进行综合评估 |
| **结构化报告** | 生成风险等级、评分、因素分析、改善建议 |
| **可视化数据** | 可选生成雷达图、仪表盘图表数据 |

**两级计算架构：**
1. **MedicalCalculator：** BMI 计算、血糖等级判定、基础风险评分
2. **Dify 风险评估工作流：** 综合医学计算器结果与问卷数据，由 DeepSeek 进行深度推理

### 6.4 健康方案生成（plan-service）

| 大模型能力 | 说明 |
|-----------|------|
| **个性化方案生成** | 基于用户画像生成饮食/运动/作息/用药四大维度方案 |
| **热量计算** | CalorieCalculator（Mifflin-St Jeor 公式）计算每日推荐热量 |
| **分阶段推送** | 后端将 Dify 阻塞响应拆分为多个 SSE 事件推送 |

### 6.5 健康趋势分析（user-service）

| 大模型能力 | 说明 |
|-----------|------|
| **趋势分析** | 分析 BMI、血糖、血压等指标的变化趋势 |
| **异常检测** | 自动检测异常波动并生成警示信息 |
| **趋势总结** | 生成文本总结 + 可视化数据点 |

**输入：** 近 30 条历史健康记录、用户基线数据。

**输出：** 各指标趋势方向（rising/falling/stable/fluctuating）、异常列表（含严重程度和建议）、文字总结。

### 6.6 打卡 AI 行为分析（checkin-service）

| 大模型能力 | 说明 |
|-----------|------|
| **行为模式识别** | 分析各类型打卡规律（规律/不稳定/需加强） |
| **异常检测** | 检测漏打卡、血糖异常等事件 |
| **改善建议** | 生成个性化打卡改善建议 |

### 6.7 资讯模块（article-service + 管理端）

| 大模型能力 | 说明 | 调用方 |
|-----------|------|--------|
| **个性化推荐重排** | AI 对 Top-N 候选文章基于用户画像重排序 | article-service |
| **资讯初稿生成** | 基于主题/关键词生成 Markdown 资讯初稿 | 管理端前端直调 Dify |

## 9. Dify 工作流开发规范

### 9.1 开始节点配置

1. **变量命名：** 所有工作流开始节点统一使用规范变量名（如 `inputs`、`query`、`user_id` 等）
2. **JSON Schema：** 将对应的 `*.schema.json` 文件内容粘贴到 Dify 开始节点的 JSON Schema 配置中
3. **变量类型选择：**
   - 简单字符串 → Text / Paragraph 类型
   - JSON 对象 → 若为顶层字段选 Object 类型聚合；若为独立入参选 Text 类型传 JSON 字符串
   - 数值 → Number 类型

### 9.2 Prompt 设计规范

1. **角色设定：** 在每个 LLM 节点系统提示词中明确专家角色
2. **输出约束：** 明确输出格式要求（JSON Schema / Markdown 格式）
3. **知识注入：** 将知识库检索结果注入 `knowledge_context` 变量
4. **安全边界：** 必须包含医疗免责声明（如"请在医生指导下执行"）

### 9.3 输出节点配置

1. **输出变量命名：** 使用有意义的输出变量名（如 `behavior_analysis`、`plan_llm_output`、`risk_assessment`）
2. **JSON 深度限制：** Dify 结束节点 JSON 深度限制为 5 层，超 5 层的嵌套结构需后端组装（参见健康方案生成的浅层输出方案）
3. **兜底处理：** 当 LLM Structured Output 解析失败时，应兜底使用普通 LLM 文本输出

### 9.4 工作流 Schema 文件管理

每个工作流在对应微服务的 `src/main/resources/dify/` 目录下维护 Schema 文件：

```
backend/
├── checkin-service/src/main/resources/dify/
│   └── checkin-analysis-input.schema.json
├── plan-service/src/main/resources/dify/
│   ├── plan-generation-input.schema.json
│   └── plan-generation-llm-output.schema.json
├── health-service/src/main/resources/dify/
│   └── risk-assessment-input.schema.json
└── article-service/src/main/resources/dify/
    └── article-recommend-input.schema.json
```

**配置 Dify 工作流时，将对应 Schema 文件内容粘贴到开始节点的 `inputs` 变量 JSON Schema 配置中。**

## 10. 开发指南

### 10.1 新增一个大模型工作流

按以下步骤在系统中新增一个大模型工作流：

#### Step 1：Dify 端创建

1. 在 Dify 平台创建新的工作流应用
2. 配置开始节点的输入变量及 JSON Schema
3. 编写 LLM 节点的系统提示词与输出约束
4. 配置知识库检索节点（如需要）
5. 配置结束节点的输出变量
6. 获取 API Key

#### Step 2：后端编写契约类

在对应微服务中创建 `*WorkflowContract.java`，参考现有契约类：

```java
// 示例：行为分析契约类
public class DifyCheckinAnalysisWorkflowContract {
    
    // 构建 Dify API 请求 payload
    public Map<String, Object> buildInputObject(/* 业务参数 */) {
        // 按工作流要求组装 inputs
    }
    
    // 解析 Dify 响应
    public BehaviorAnalysisVO parseOutput(Map<String, Object> outputs) {
        // 从 outputs 中提取所需字段
    }
}
```

#### Step 3：编写 Schema 文件

在 `src/main/resources/dify/` 下创建对应的 `*.schema.json` 文件。

#### Step 4：配置环境变量

添加 `DIFY_*_API_KEY` 和相关配置。

#### Step 5：在 Service 中集成调用

```java
@Service
public class CheckinAIService {
    
    @Autowired
    private DifyClient difyClient;
    
    @Value("${dify.checkin.api-key}")
    private String apiKey;
    
    public BehaviorAnalysisVO generateSummary(userId, dateRange) {
        Map<String, Object> inputs = workflowContract.buildInputObject(userId, dateRange);
        DifyResponse response = difyClient.runWorkflow(apiKey, inputs, userId, "blocking");
        return workflowContract.parseOutput(response.getOutputs());
    }
}
```

#### Step 6：编写契约查询 API

提供 `GET /api/v1/xxx/dify-workflow-spec` 接口，返回工作流入参/出参规范，便于前后端联调（参考现有 `*WorkflowContractService` 实现）。

### 10.2 调试技巧

1. **Dify 调试：** 先在 Dify 控制台使用「运行」功能单独测试工作流
2. **日志排查：** 查看 Dify 工作流运行日志，检查各节点输出
3. **后端调试：** 在 `DifyClient` 中添加请求/响应日志，确认 payload 与返回结构
4. **契约验证：** 调用 `GET /api/v1/xxx/dify-workflow-spec` 确认契约一致性

### 10.3 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| Dify 返回 401 | API Key 错误 | 检查环境变量中的 `DIFY_*_API_KEY` |
| JSON 解析失败 | 输出结构不匹配 | 对比 Schema 文件与 Dify 实际返回，更新契约类 |
| 嵌套超过 5 层 | Dify JSON 深度限制 | 改用浅层输出 + 后端组装（参见健康方案生成方案）|
| SSE 连接中断 | 超时设置过短 | 增加 `DifyClient` 中的超时时间 |
| 知识库检索无结果 | Milvus 未同步或 Embedding 不一致 | 检查 Milvus 连接与数据导入状态 |
