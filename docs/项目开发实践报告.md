## 3. 实践内容和实践过程（Project Content and Process）

### 3.1 概述（Overview）

#### 3.1.1 项目简介

**糖尿病智能助手**是一套基于微服务架构的糖尿病预治一体化管理平台，面向糖尿病患者、高危人群、内分泌科医生、健康管理师及系统管理员，提供"科普教育 → 风险评估 → AI 问诊 → 健康方案 → 日常打卡 → 行为分析 → 资讯推荐"的完整健康管理闭环。

系统采用 **双前端 + 微服务后端 + AI 编排层** 的技术架构：

- **用户端前端**（`frontend/`）：面向普通用户的健康管理门户，涵盖 8 大业务模块共 20+ 页面；
- **管理端前端**（`admin-frontend/`）：面向管理员的运营后台，支持资讯管理、视频管理、统计看板与审计日志；
- **后端微服务集群**（`backend/`）：8 个业务微服务 + 1 个 API 网关 + 1 个公共模块，按领域拆分、数据库隔离；
- **AI 能力层**：通过 Dify 平台编排 10+ 个工作流，对接 DeepSeek 大模型与 Milvus 医学知识库。

### 3.3 系统设计（System Design）

#### 3.3.1 总体架构

系统采用**四层分层架构**，自顶向下依次为：

```
表现层（Vue 3 双前端）
    ↓
接入层（Nginx + Spring Cloud Gateway）
    ↓
业务服务层（8 个微服务 + common 公共模块）
    ↓
AI 能力层（Dify 工作流 + DeepSeek 大模型）
    ↓
数据存储层（MySQL / Redis / MinIO / Milvus / Kingbase）
```

**核心设计原则**：

1. **微服务拆分**：按业务领域（用户、健康、打卡、问诊、方案、资讯等）独立部署，数据库隔离；
2. **API 网关统一入口**：Gateway 负责 `/api/v1/**` 和 `/api/v2/**` 路由分发，支持 API 版本演进；
3. **契约驱动 AI 集成**：每个 Dify 工作流对应 Java Contract 类与 JSON Schema，确保前后端与 AI 层数据格式一致；
4. **优雅降级**：Dify API Key 缺失时，各服务自动切换本地规则引擎或模板回复，保障基本可用性；
5. **审计隔离**：业务数据存 MySQL，审计日志独立写入金仓数据库，异步 fire-and-forget 模式不阻塞主流程。

#### 3.3.3 数据库设计

MySQL 按微服务划分为 9 个逻辑库，各服务仅访问自身数据库，通过 HTTP 内部接口获取跨域数据。主要实体包括：

- **用户域**：用户账户、健康档案、消息通知、干预策略
- **打卡域**：食物/用药/运动/血糖打卡记录、成就徽章、提醒规则
- **问诊域**：AI 医生、咨询会话、消息时间线
- **方案域**：健康方案、热量计算记录
- **资讯域**：文章、分类、收藏、推荐缓存

向量数据存储于 Milvus 的两个 Collection：`diabetes_knowledge`（医学知识 RAG）和 `article_knowledge`（资讯语义检索）。

### 3.4 编码（Coding）

#### 3.4.1 开发流程

项目采用**自底向上、契约先行**的编码策略：

```
数据契约定义 → 数据库建表 → 后端 Service/Controller → 前端 API/页面 → 联调测试
```

具体步骤：

1. **公共模块先行**：在 `backend/common` 中实现 `DifyClient`、`JwtUtil`、`MinioStorageService`、各服务 HTTP 客户端等共享能力；
2. **契约类与 Schema**：每个 Dify 工作流先定义 `*WorkflowContract.java` 和 `*.schema.json`，再编写 Service 调用逻辑；
3. **服务独立开发**：各微服务按 Controller → Service → Mapper → Entity 分层实现，通过内部 HTTP 接口通信；
4. **前端 API 层封装**：`frontend/src/api/` 按业务模块组织，统一 Axios 拦截器处理 Token 与错误；
5. **Composable 复用**：将消息中心、打卡提醒、语音输入等逻辑封装为 Vue Composable，提高代码复用率；
6. **字段规范化**：通过 `normalize.js` 处理后端 camelCase 与前端 snake_case 的自动转换。

#### 3.4.2 后端编码要点

**Dify 工作流调用**（以问诊服务为例）：

```java
// 1. 构建工作流输入（契约类保证字段完整）
ConsultationWorkflowInputs inputs = ConsultationWorkflowInputs.builder()
    .userMessage(message)
    .healthContext(healthContext)
    .knowledgeContext(knowledgeContext)
    .conversationHistory(history)
    .build();

// 2. 通过 DifyClient 调用（支持 blocking / streaming）
DifyWorkflowResponse response = difyClient.runWorkflow(
    workflowApiKey, inputs.toMap(), DifyRunMode.BLOCKING);

// 3. 解析输出并持久化
ConsultationWorkflowOutputs outputs = ConsultationWorkflowOutputs.from(response);
```

**服务间通信**：通过 `common` 模块中的 Typed Client（如 `HealthServiceClient`、`HomeServiceClient`）进行同步 HTTP 调用，无消息总线。

**优雅降级**：当 `DIFY_API_KEY` 未配置时，`ConsultationService` 自动生成模板回复，确保功能基本可用。

#### 3.4.3 前端编码要点

**路由模块化**：`frontend/src/router/modules/` 按业务拆分路由配置，通过 `meta.requiresAuth` 控制登录鉴权。

**SSE 流式消费**（以 AI 科普问答为例）：

```javascript
// 使用 fetch + ReadableStream 消费 SSE
const response = await fetch('/api/v1/chat/stream', { method: 'POST', body });
const reader = response.body.getReader();
// 逐块解析 data: 事件，实时更新 Markdown 渲染
```

**Composable 封装**：`useVoiceInput` 封装 MediaRecorder 录音 → 上传 STT → 返回文本的完整流程；`useMessageCenter` 封装消息列表、已读标记与轮询逻辑。

## 4. 实践总结（Conclusion）

### 4.1 经验与收获

1. **微服务架构的实践认知**：通过按领域拆分 8 个微服务并实施数据库隔离，深刻理解了"高内聚、低耦合"的设计原则。服务间通过 HTTP 客户端通信虽然简单直接，但在调试和链路追踪上需要额外投入；公共模块 `common` 的抽取有效避免了代码重复。

2. **AI 工程化的落地方法**：契约驱动（Contract-First）的 Dify 工作流集成模式是本项目的核心创新。先定义 JSON Schema 和 Java Contract 类，再编写调用逻辑，大幅降低了 AI 层与业务层之间的集成风险。RAG 链路（Milvus 检索 → 上下文注入 → Dify 生成）的实现让我理解了检索增强生成的完整流程。

3. **优雅降级的设计思维**：在 AI 服务不可用时自动切换本地规则引擎或模板回复，保障了系统的基本可用性。这一设计在开发和测试阶段尤为实用——无需依赖外部 Dify 实例即可完成大部分功能验证。

4. **测试驱动的质量保障**：JaCoCo 和 Vitest 的覆盖率门禁机制迫使开发者在编码时同步考虑测试用例，consultation-service 达到 100% 覆盖率的过程也帮助发现了多处边界条件遗漏。Docker 化的测试环境确保了跨机器的一致性。

5. **全栈开发的综合能力**：从数据库设计、后端 API、前端页面到 AI 工作流编排、容器化部署，本次实践覆盖了现代 Web 开发的完整技术栈，对软件工程的全生命周期有了系统性的认识。

### 4.2 问题与教训

1. **文档与代码同步滞后**：部分早期文档（如 `backend/README.md`、`前后端联调调用流程.md`）未及时更新，仍描述已废弃的占位实现，给后续开发者造成困惑。教训：**代码变更应同步更新关联文档**，或将文档生成纳入 CI 检查。

2. **首页内容 API 未完成**：`GET /api/v1/home/content` 仍返回 501，说明模块优先级管理不够清晰。教训：应在迭代计划中明确各模块的完成标准，避免"半成品"长期遗留。

3. **字段命名不一致的隐性成本**：后端 Java 使用 camelCase、部分 Dify 工作流使用 snake_case，需要 `normalize.js` 做双向转换。教训：**应在项目初期统一命名规范**，并在 API 契约中强制执行。

4. **服务间同步调用的局限性**：当前所有跨服务通信均为同步 HTTP，在健康干预编排等需要调用多个服务的场景下，响应延迟叠加明显。未来可考虑引入消息队列实现异步解耦。

5. **国产数据库适配成本**：金仓 KingbaseES 在 SQL 方言、驱动兼容性上与 MySQL 存在差异，审计服务的适配花费了额外时间。教训：选用国产数据库时应提前做 POC 验证。

### 4.3 感悟与展望

本次实践让我深刻体会到，**AI 不是独立的技术模块，而是需要深度融入业务流的赋能手段**。将大模型能力通过 Dify 工作流嵌入到风险评估、健康方案、行为分析等具体场景中，比单纯做一个"聊天机器人"更有价值。

在工程层面，**契约驱动 + 测试门禁 + 容器化部署**的组合为项目质量提供了坚实保障。特别是 Docker Compose 一键启动 17+ 容器的体验，极大降低了团队协作和环境一致性成本。

未来可在以下方向继续深化：

- 完成食物图像识别工作流的后端对接，丰富打卡场景；
- 落地用户健康画像 Agent，实现跨模块数据的智能聚合；
- 引入消息队列（如 RabbitMQ）优化服务间异步通信；
- 完善首页内容管理与科普视频的前端展示；
- 将压测脚本集成到 CI/CD 流水线，建立性能回归基线。
