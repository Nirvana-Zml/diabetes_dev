# 前后端联调调用流程

> 约束：不修改 `views/Home`、`views/Assistant`、`views/Consultation`；不实现 `home-service`、`consultation-service`；Dify 仅前端占位。

## 1. 架构总览

```
浏览器 (Vue3 + Element Plus)
    │  Authorization: Bearer {JWT}
    ▼
Gateway :8099  /api/v1/**
    ├── user-service:8081      认证 / 个人中心
    ├── health-service:8083    风险预测 / 健康档案
    ├── checkin-service:8084   打卡 / 打卡管理
    ├── plan-service:8086      健康方案（SSE）
    ├── article-service:8087   健康资讯
    └── (占位，不联调) home-service / consultation-service
```

前端统一入口：`frontend/src/utils/request.js`  
字段适配：`frontend/src/utils/normalize.js`（后端 camelCase ↔ 前端 snake_case）

## 2. 启动联调

```bash
# 后端（需 Redis：打卡/资讯缓存）
docker compose up -d mysql redis
docker compose up -d --build gateway user-service health-service checkin-service plan-service article-service

# 前端（VITE_USE_MOCK=false）
cd frontend && npm install && npm run dev
```

测试账号：`testuser` / `123456`（见 `db/init.sql`）

## 3. 模块调用流程

### 3.1 登录认证（user-service）

| 步骤 | 前端 | 后端 |
|------|------|------|
| 1 | `views/Login` → `api/auth.js` `login()` | `POST /api/v1/auth/login` |
| 2 | `saveTokens()` 存 `access_token` | 返回 `TokenResponse` |
| 3 | 后续请求自动带 `Authorization` | `JwtAuthInterceptor` 校验 |

### 3.2 糖尿病风险预测（health-service）

| 步骤 | 前端 | 后端 |
|------|------|------|
| 1 | `views/HealthEvaluation` 填表 | — |
| 2 | `api/risk.js` `assessRisk()` 映射字段 → `POST /risk/assess` | `RiskAssessmentController` |
| 3 | — | `MedicalCalculator` 本地计算 + **Dify 工作流**（可选） |
| 4 | `normalizeRiskResult()` 解析展示 | 写入 `RISK_ASSESSMENTS` 等表 |
| 5 | 历史 Tab → `getRiskHistory()` | `GET /risk/history` |

**不经过前端 Dify**：`api/dify.js` 的 `difyRiskAssess` 仅在 `VITE_USE_MOCK=true` 时由 `risk.js` 使用。

### 3.3 健康方案（plan-service）

| 步骤 | 前端 | 后端 |
|------|------|------|
| 1 | `views/LivingPlans` 点击生成 | — |
| 2 | `api/plan.js` `generatePlan()` | `POST /plan/generate`（SSE） |
| 3 | `utils/sse.js` `fetchBackendSSE` 解析 `stage_*` 事件 | `PlanService` → **Dify 工作流** |
| 4 | `getLatestPlan()` / `getPlanHistory()` | `GET /plan/latest`、`/plan/history` |

Mock 模式：`generatePlan` 走 `difyPlanGenerate` 占位。

### 3.4 生活打卡（checkin-service）

| 步骤 | 前端 | 后端 |
|------|------|------|
| 1 | `views/CheckinRecords` 提交打卡 | `api/checkin.js` `createCheckin()` |
| 2 | 映射 `checkin_type` → `checkinType` | `CheckinController` → MyBatis 写库 |
| 3 | 今日状态 / 统计 / 成就 | `GET /checkin/today`、`/stats`、`/achievements` |

### 3.5 打卡分析（checkin-service 管理接口）

| 步骤 | 前端 | 后端 |
|------|------|------|
| 1 | `views/CheckinAnalysis` 选择周期 | `buildDateRange()` 转 `startDate/endDate` |
| 2 | `api/checkinManagement.js` | `GET /checkin-management/stats|trends|ai-summary` |
| 3 | AI 总结展示 Markdown | 后端 `CheckinMgmtService` → **Dify**（可选降级） |

### 3.6 健康资讯（article-service）

| 步骤 | 前端 | 后端 |
|------|------|------|
| 1 | `views/HealthInfo` 列表/搜索 | `GET /articles`、`/articles/search` |
| 2 | 详情 / 收藏 | `GET /articles/{id}`、`POST /articles/{id}/favorite` |
| 3 | 分类筛选 | 前端 `category` 字符串 ↔ 后端 `CATEGORY` 整型映射 |

### 3.7 个人中心（user-service + health-service）

| 功能 | 前端 API | 后端服务 |
|------|----------|----------|
| 资料 / 密码 / 隐私 | `api/user.js` | user-service `/user/*` |
| 健康档案读写 | `api/healthRecord.js` | health-service `/health-records/*` |
| 就诊记录 | `getUserConsultations()` | **仅占位 Mock**（咨询未实现） |
| 健康趋势 / 预警 | `getHealthTrendSummary()` 等 | **仅占位 Mock**（待 Dify） |

### 3.8 占位模块（不联调后端）

| 模块 | 前端页面 | API 文件 | 说明 |
|------|----------|----------|------|
| 科普首页 | `views/Home`（禁止改） | `api/home.js` | 始终 `mockFn`，不请求 home-service |
| AI 助手 | `views/Assistant`（禁止改） | `api/chat.js` → `dify.js` | Dify 占位 SSE |
| 医生咨询 | `views/Consultation`（禁止改） | `api/consultation.js` | 始终 Mock，不请求 consultation-service |

## 4. 环境变量

| 变量 | 说明 |
|------|------|
| `VITE_USE_MOCK=false` | 启用真实后端（占位模块仍用 mockFn） |
| `VITE_API_BASE=/api/v1` | 与 Gateway 路由一致 |
| `VITE_PROXY_TARGET` | Vite 代理网关（Docker 内 `http://gateway:8099`） |
| `VITE_USER_SERVICE_TARGET` | 登录/注册直连 user-service（Docker 内 `http://user-service:8081`，绕过 Gateway 代理兼容问题） |
| `JWT_SECRET` | 前后端 JWT 一致 |
| `DIFY_*_API_KEY` | 各服务 Dify 工作流密钥（后端调用） |
| `REDIS_PASSWORD` | Redis 密码（Docker 内 `redis:6379`，宿主机 `localhost:6380`） |

业务缓存 Key 前缀为 `diabetes:`，可与 Dify 共用同一 Redis 实例。

## 5. 异常处理

- `request.js` 拦截器：`code !== 200` → `Promise.reject(Error message)`
- 页面层 `try/catch` + `ElMessage.error`
- 401：Token 无效，需重新登录
