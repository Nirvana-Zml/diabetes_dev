# 前后端测试报告

> **文档版本**：2026-07-02  
> **适用范围**：用户端前端（`frontend/`）、后端微服务（`backend/`）  
> **后端测试环境**：Docker 容器内 Maven，挂载目录 `docker/maven/`

---

## 1. 概述

本项目采用分层单元测试策略：

| 层级 | 框架 | 覆盖率工具 | 说明 |
|------|------|------------|------|
| 后端 | JUnit 5 + Spring Boot Test + Mockito | JaCoCo 0.8.12 | 在 Docker 中运行，不依赖宿主机 JDK/Maven |
| 用户端前端 | Vitest 2 + Vue Test Utils + happy-dom | @vitest/coverage-v8 | 在宿主机 Node.js 环境运行 |
| 管理端前端 | — | — | `admin-frontend/` 暂未配置自动化测试 |

后端 **8 个业务微服务中，除 `gateway` 外均有单元测试**（`user-service`、`home-service`、`health-service`、`checkin-service`、`consultation-service`、`plan-service`、`article-service`）；另含公共库 `common` 的测试。合计 **1067** 个用例；最近一次全量 `mvn test` 全部通过，耗时约 **21 分钟**。

用户端前端共 **25** 个测试文件（`.spec.js` / `.test.js`），覆盖 API 层、工具函数、组件、路由、Store 及页面冒烟逻辑。

## 2. 后端测试

### 2.1 Docker Maven 环境

后端测试统一在 Docker 中执行，相关文件：

```
docker-compose.maven.yml          # Maven 工具容器编排
docker/maven/
├── settings.xml                    # Maven 配置（阿里云镜像）
├── repository/                     # 依赖缓存（挂载持久化）
├── mvn.ps1                         # 通用 Maven 命令封装
└── run-test.ps1                    # 单模块测试快捷脚本
```

容器配置要点（`docker-compose.maven.yml`）：

- **镜像**：`maven:3.9-eclipse-temurin-17-alpine`
- **工作目录**：`/app`（挂载 `./backend`）
- **依赖缓存**：`./docker/maven/repository` → `/root/.m2/repository`
- **Maven 配置**：`./docker/maven/settings.xml` → `/root/.m2/settings.xml`

> **注意**：不要直接执行 `docker compose run maven -version`，Compose 会把 `-v` 解析为卷参数。请使用下文封装脚本。

### 2.2 如何运行测试

#### 全量测试（所有模块）

在项目根目录执行：

```powershell
.\docker\maven\mvn.ps1 test -B
```

#### 单模块测试

```powershell
# 仅运行 user-service 测试
.\docker\maven\mvn.ps1 test -pl user-service -B

# 带依赖编译（推荐首次或代码变更后）
.\docker\maven\run-test.ps1 -Service user-service

# 运行 test 并执行 JaCoCo 门禁（verify 阶段）
.\docker\maven\run-test.ps1 -Service user-service -Verify
```

#### 直接使用 Docker Compose

```powershell
docker compose -f docker-compose.maven.yml run --rm --entrypoint sh maven -c "mvn -pl article-service test -B"
```

#### 查看覆盖率报告

测试完成后，各模块 HTML 报告位于：

```
backend/<module>/target/site/jacoco/index.html
```

在浏览器中打开即可查看包级、类级明细。

#### 覆盖率门禁（`mvn verify`）

部分模块在 `pom.xml` 中配置了 JaCoCo `check` 规则，`verify` 阶段会强制校验：

| 模块 | 门禁包 | 行/分支最低要求 |
|------|--------|-----------------|
| user-service | `service`、`controller`、`export` | 100% |
| user-service | `config`、`entity`、`dto`、`util` | 分支 ≥ 80% |
| checkin-service | 全部业务包 | 指令 + 分支 100% |
| plan-service | `service` | 行 + 分支 100% |
| article-service | `controller`、`entity`、`service` | 行 + 分支 100% |
| article-service | `config`、`dify`、`milvus` | 行 + 分支 ≥ 80% |

日常开发可先用 `mvn test` 快速反馈；合并前建议对改动模块执行 `mvn verify` 确保门禁通过。

### 2.5 测试设计特点

1. **外部依赖 Mock**：Dify、Milvus、Redis、MinIO、邮件等均通过 Mockito 或 Stub 隔离，测试可在离线环境运行。
2. **工作流契约测试**：各 `*WorkflowContractTest` 校验 Dify 输入/输出 JSON 结构，防止工作流变更导致集成失败。
3. **降级路径覆盖**：API Key 未配置、网络异常、工作流失败等场景均有对应用例，验证本地规则/模板降级。
4. **反射/私有方法测试**：部分复杂服务（如 `ArticleRecommendServiceReflectionTest`）通过反射覆盖私有辅助逻辑，补齐分支覆盖率。

## 3. 用户端前端测试

### 3.1 环境要求

- Node.js 18+（推荐 LTS）
- 在 `frontend/` 目录安装依赖：`npm install`

### 3.2 如何运行测试

```bash
cd frontend

# 交互模式（文件变更自动重跑）
npm test

# 单次运行全部用例
npm test -- --run

# 生成覆盖率报告
npm run test:coverage -- --run
```

覆盖率报告输出：

- 终端：text 摘要

- JSON：`frontend/coverage/coverage-summary.json`

`frontend/vite.config.js` 中配置了 Vitest 阈值（未达标时 `test:coverage` 失败）：

| 指标 | 最低要求 |
|------|----------|
| statements（语句） | 90% |
| branches（分支） | 75% |
| functions（函数） | 80% |
| lines（行） | 90% |

统计范围：`src/**/*.{js,vue}`，排除测试文件本身。

> 请在本地执行 `npm run test:coverage -- --run` 获取最新覆盖率数值；CI 或本地 Node 环境就绪后，报告将写入 `frontend/coverage/`。

### 3.5 主要测试说明

#### API 层（`src/api/__tests__/`）

- **`auth.test.js`**：登录、注册、Token 存储与请求头注入。
- **`request.spec.js`**：Axios 实例、401 跳转、错误消息提取。
- **`services.spec.js`**：各业务 API 模块导出与 URL 拼装。
- **`article.spec.js` / `chat-live.spec.js` / `plan-live.spec.js`**：资讯、聊天、方案相关接口（Mock `fetch`/Axios）。

#### 工具函数（`src/utils/__tests__/`）

- **`sse.spec.js`**：Server-Sent Events 解析、断线重连逻辑。
- **`auth.test.js`**：Token 读写、过期判断。
- **`normalize.spec.js`**：后端响应字段归一化。

#### 组件测试（`src/components/__tests__/`）

- **`AiChatDialog.interactions.spec.js`**：AI 对话弹窗发送、流式展示、关闭行为。
- **`MarkdownContent.spec.js`**：Markdown 渲染与 XSS 相关处理。

#### 页面冒烟（`src/views/__tests__/views-smoke.spec.js`）

对用户端主要 Vue 页面进行 **shallowMount** 冒烟测试：挂载组件、探测模板事件处理器、验证关键 computed/method 可调用。该文件是前端覆盖率的主要贡献者，确保各业务页面在重构时不因语法或导入错误而整体崩溃。

#### 路由与状态（`src/router/`、`src/stores/`）

- 未登录访问受保护路由时的重定向。
- User Store 登录态、用户信息缓存与清除。

### 3.6 测试设计特点

1. **happy-dom 环境**：轻量 DOM，适合 Vue 组件单元测试，启动快。
2. **广泛 Mock**：`vue-router`、`element-plus`、API 模块在测试中统一 Mock，避免依赖后端服务。
3. **冒烟 + 精准单测结合**：复杂页面用 smoke 保覆盖，核心逻辑（API、工具、对话框）用精细断言。

## 5. 常见问题

### Q1：`docker compose run maven` 报错或参数被误解析？

使用项目提供的 `.\docker\maven\mvn.ps1` 或 `run-test.ps1`，不要省略 `-f docker-compose.maven.yml` 和 `--entrypoint sh maven -c "..."` 形式。

### Q2：首次 Docker 测试很慢？

依赖会下载到 `docker/maven/repository/`，第二次起显著加速。必要时可只测单模块：`-pl user-service`。

### Q3：Surefire 日志出现 `Surefire is going to kill self fork JVM`？

部分测试类调用了 `System.exit(0)`（如 Milvus 相关测试），Surefire 会等待 30 秒后强制结束子进程。此为警告，**不影响测试结果**。

### Q4：前端 `test:coverage` 未达阈值？

优先查看 `frontend/coverage/index.html` 中覆盖率最低的 `src/views/` 页面，或在 `views-smoke.spec.js` 中补充对应路由的冒烟用例。

### Q5：`mvn verify` 与 `mvn test` 的区别？

- `test`：运行单元测试并生成 JaCoCo 报告。
- `verify`：额外执行 JaCoCo `check` 门禁；未达模块配置的最低覆盖率时会构建失败。
