> **文档版本**：2026-07-02（修订 4）

## 1. 概述

本项目采用分层单元测试策略：

| 层级 | 框架 | 覆盖率工具 | 运行环境 |
|------|------|------------|----------|
| 后端 | JUnit 5 + Spring Boot Test + Mockito | JaCoCo 0.8.12 | `docker-compose.maven.yml`（`maven:3.9-eclipse-temurin-17-alpine`） |
| 用户端前端 | Vitest 2 + Vue Test Utils + happy-dom | @vitest/coverage-v8 | `docker-compose.yml` → `frontend`（`node:20-alpine`） |
| 管理端前端 | — | — | `admin-frontend/` 暂未配置自动化测试 |

### 后端

7 个业务微服务 **均有单元测试**：`user-service`、`home-service`、`health-service`、`checkin-service`、`consultation-service`、`plan-service`、`article-service`；公共库 `common` 亦有测试。**`gateway` 为纯路由网关，不纳入单元测试范围。**

全量后端测试（排除 gateway）命令：`.\docker\maven\mvn.ps1 test -pl '!gateway' -B`  
最近一次实测（user-service 更新后估算）：**1212** 用例，**BUILD SUCCESS**。

### 前端

测试源码位于 `frontend/src/**/__tests__/`，共 **25** 个文件、**86** 条用例。  
最近一次 Docker 实测（`npm run test:coverage -- --run`）：**86 通过 / 0 失败**，覆盖率 **全部达标**，耗时约 **12 秒**。

### 2.5 测试设计特点

1. **外部依赖 Mock**：Dify、Milvus、Redis、MinIO、邮件等均通过 Mockito 或 Stub 隔离，测试可在离线环境运行。
2. **工作流契约测试**：各 `*WorkflowContractTest` 校验 Dify 输入/输出 JSON 结构，防止工作流变更导致集成失败。
3. **降级路径覆盖**：API Key 未配置、网络异常、工作流失败等场景均有对应用例，验证本地规则/模板降级。
4. **反射/私有方法测试**：部分复杂服务（如 `ArticleRecommendServiceReflectionTest`）通过反射覆盖私有辅助逻辑，补齐分支覆盖率。

## 3. 用户端前端测试

### 3.1 Docker Node 环境

前端测试使用 `docker-compose.yml` 中的 **`frontend`** 服务（与开发环境同一镜像与挂载）：

日常开发时该服务默认执行 `npm install && npm run dev`；**跑测试时**用 `docker compose run` 覆盖命令，无需启动 gateway 等依赖：

```powershell

docker compose run --rm --no-deps frontend sh -c "npm install && npm run test -- --run"

# 生成覆盖率（存在失败用例时可能无法通过门禁阈值）
docker compose run --rm --no-deps frontend sh -c "npm run test:coverage -- --run"
```

也可在已运行的 `diabetes-frontend` 容器内执行：

```powershell
docker exec -it diabetes-frontend npm run test -- --run
```

### 3.2 测试目录与命名规范

所有前端测试文件放在 **`src/**/__tests__/`** 目录下，与被测源码同级或就近放置。文件名遵循以下约定：

| 后缀 | 用途 | 示例 |
|------|------|------|
| `*.spec.js` | 模块/组件/页面规格测试（主流程、交互、冒烟） | `views-smoke.spec.js`、`router.spec.js` |
| `*.test.js` | 轻量单元测试（纯函数、小模块） | `auth.test.js`、`delay.test.js` |

**规则摘要**：

- 目录名固定为 **`__tests__`**（双下划线，复数）。
- 文件名应体现被测对象，如 `article.spec.js` 测 `api/article.js`，`user.spec.js` 测 `stores/user.js`。
- 同目录下 `.spec.js` 与 `.test.js` 均可混用；新增测试优先与邻近文件保持一致后缀。
