# 开发指南

本文档面向参与「糖尿病智能助手」开发的工程师，说明环境搭建、项目结构、本地调试、测试与 AI 集成方式。

> 部署步骤亦见 [README.md](./README.md)；业务操作见 [使用说明.md](./使用说明.md)。

---

## 1. 开发环境

### 1.1 版本要求

| 工具 | 版本 |
|------|------|
| JDK | 17 |
| Maven | 3.9+ |
| Node.js | 20.x |
| Docker | 24.0+ |
| Docker Compose | 2.20+ |

### 1.2 外部依赖（宿主机）

以下服务不在 `docker-compose.yml` 内，需单独安装：

| 服务 | 默认地址 | 用途 | 必须性 |
|------|----------|------|--------|
| Dify | `http://localhost:58080` | AI 工作流编排 | AI 功能必须 |
| Ollama | `http://localhost:11434` | Embedding（如 `qwen3-embedding:0.6b`） | Milvus 语义检索推荐 |
| 阿里云 DashScope | API Key | Fun-ASR 语音识别、资讯 TTS | 语音功能推荐 |

Docker 容器访问宿主机服务时使用 `host.docker.internal`（compose 已配置 `extra_hosts`）。

## 2. 配置说明

### 2.1 根目录 `.env`

```bash
cp .env.example .env
```

**必填项（生产/联调）：**

```bash
MYSQL_ROOT_PASSWORD=...
REDIS_PASSWORD=...
JWT_SECRET=...                    # 至少 32 字符，所有 backend 服务须一致
MINIO_PASSWORD=...
KINGBASE_PASSWORD=...
DIFY_BASE_URL=http://host.docker.internal:58080
DIFY_INTERNAL_KEY=...             # Dify 回调 health-service 内部 API
# 各工作流 API Key — 见 .env.example 中 DIFY_*_API_KEY
```

**可选但常用：**

```bash
DASHSCOPE_API_KEY=...             # STT / TTS
MAIL_USERNAME=...                 # QQ 邮箱 SMTP（注册验证码）
MAIL_PASSWORD=...                   # QQ 邮箱授权码
MILVUS_ENABLED=true
EMBEDDING_PROVIDER=openai
EMBEDDING_BASE_URL=http://host.docker.internal:11434
EMBEDDING_MODEL=qwen3-embedding:0.6b
```

完整变量说明见 [`.env.example`](./.env.example)。

### 2.2 前端环境变量

**用户端** `frontend/.env`：

```bash
cp frontend/.env.example frontend/.env
VITE_USE_MOCK=false                # 联调真实后端时设为 false
VITE_ADMIN_PORTAL_URL=http://localhost:81
```

**管理端** `admin-frontend/.env`：

```bash
cp admin-frontend/.env.example admin-frontend/.env
VITE_USE_MOCK=false
VITE_USER_PORTAL_URL=http://localhost
```

Vite dev 将 `/api/v1`、`/api/v2` 代理到 Gateway `http://localhost:8099`；管理端额外代理 `/dify-proxy` 到 Dify。

## 3. 启动方式

### 3.1 Docker Compose（推荐）

```bash
docker compose up -d
docker compose ps
docker compose logs -f gateway user-service
```

首次启动 MySQL 自动执行 [`db/init.sql`](./db/init.sql)（分库 + 测试数据）；Kingbase 执行 [`db/kingbase_audit_init.sql`](./db/kingbase_audit_init.sql)。

### 3.2 仅基础设施 + 本地 IDE 调试

```bash
docker compose up -d mysql redis minio milvus kingbase nginx gateway
# 或在 IDE 中单独启动各 *-service JAR
```

### 3.3 本地前端

```bash
cd frontend && npm install && npm run dev      # :5173
cd admin-frontend && npm install && npm run dev # :5174
```

### 3.4 本地后端

```bash
cd backend
mvn clean package -DskipTests
java -jar gateway/target/gateway-1.0.0-SNAPSHOT.jar
java -jar user-service/target/user-service-1.0.0-SNAPSHOT.jar
# ... 按需启动其余服务
```

## 4. 后端架构

### 4.1 微服务一览

| 服务 | 端口 | 数据库 | 职责 |
|------|------|--------|------|
| gateway | 8099 | — | 路由 `/api/v1`、`/api/v2` |
| user-service | 8081 | DIABETES_USER | JWT 认证、用户/管理员、消息中心、健康趋势 AI、主动干预 |
| home-service | 8082 | DIABETES_RESOURCES | 首页、科普问答 SSE、Fun-ASR、视频 CRUD |
| health-service | 8083 | DIABETES_HEALTH | 健康档案、风险评估、Dify 回调内部 API |
| checkin-service | 8084 | DIABETES_CHECKIN | 打卡、行为分析、食物识别、提醒 |
| consultation-service | 8085 | DIABETES_CONSULTATION / DIABETES_DOCTOR | AI 问诊 |
| plan-service | 8086 | DIABETES_PLAN | 健康方案（Dify blocking → SSE 推送） |
| article-service | 8087 | DIABETES_ARTICLE | 资讯、推荐、TTS |
| audit-service | 8088 | KingbaseES | 审计日志 |

MySQL 分库定义见 [`db/init.sql`](./db/init.sql)。

### 4.2 公共模块 `backend/common`

- `DifyClient` — Dify Workflow / 文件上传 HTTP 客户端
- `DifyJsonSchema` — 工作流 Schema 构建
- `MinioStorageService` — 对象存储
- `AuditServiceClient` — 跨服务审计上报

### 4.3 API 约定

- 统一前缀：`/api/v1`（部分 v2 在 consultation 等模块）
- 认证：`Authorization: Bearer <JWT>`
- 响应：`{ "code": 200, "data": ..., "message": "..." }`
- Dify 工作流契约查询：各服务 `GET .../dify-workflow-spec`

### 4.4 Dify 工作流契约类

| 工作流 | 契约类 | Schema 文件 |
|--------|--------|-------------|
| 打卡行为分析 | `DifyCheckinAnalysisWorkflowContract` | `checkin-service/.../dify/*.schema.json` |
| 健康方案 | `DifyPlanWorkflowContract` | `plan-service/.../dify/*.schema.json` |
| 风险评估 | `DifyRiskAssessmentWorkflowContract` | `health-service/.../dify/*.schema.json` |
| 资讯推荐 | `DifyArticleRecommendWorkflowContract` | `article-service/.../dify/*.schema.json` |
| 健康趋势 | `DifyHealthTrendWorkflowContract` | 契约类内 Schema |
| AI 问诊 | `DifyConsultationWorkflowContract` | 契约类内 Schema |
| 科普问答 | `DifyQaChatContract` | 契约类内 Schema |
| 食物识别 | `DifyFoodRecognitionWorkflowContract` | 契约类内 Schema |
| 资讯初稿 | `DifyArticleDraftWorkflowContract` | 契约类内 Schema |

汇总文档：[docs/Dify工作流数据契约.md](./docs/Dify工作流数据契约.md)。

## 5. 前端架构

### 5.1 用户端 `frontend/`

```text
src/
├── api/           # Axios 封装，按模块划分
├── views/         # 页面（Home、CheckinRecords、Assistant 等）
├── composables/   # 可复用逻辑（消息中心、语音输入等）
├── stores/        # Pinia 状态
├── router/        # 路由与登录守卫
└── mock/          # VITE_USE_MOCK=true 时使用
```

移动端优先布局，底部导航：首页、生活打卡、资讯、AI 助手、我的。

### 5.2 管理端 `admin-frontend/`

路由：首页统计、数据统计、资讯管理、视频管理、审计日志。资讯 AI 初稿通过 `/dify-proxy` 直连 Dify SSE。

## 7. AI 集成开发

### 7.1 新增 / 修改 Dify 工作流

1. 在 Dify 控制台编排工作流，按契约配置开始节点变量与结束节点输出
2. 在后端新增 `*WorkflowContract.java`，定义 `buildInputObject`、`inputJsonSchema`、`outputJsonSchema`
3. 可选：在 `src/main/resources/dify/` 添加 `.schema.json`
4. 暴露 `GET .../dify-workflow-spec` 便于联调
5. 更新 [docs/Dify工作流数据契约.md](./docs/Dify工作流数据契约.md)

### 7.2 Milvus 知识库

科普问答 RAG 依赖 `diabetes_knowledge` Collection，导入步骤见 [docs/Milvus医学知识库落地指南.md](./docs/Milvus医学知识库落地指南.md)。

### 7.3 本地公网穿透（可选）

食物识别与语音识别需外部服务通过 **公网 HTTP** 拉取 MinIO 资源：

```bash
ngrok http 9000
# .env 中设置：
# STT_AUDIO_PUBLIC_BASE_URL=https://xxxx.ngrok-free.app
# DIFY_FOOD_RECOGNITION_IMAGE_PUBLIC_BASE_URL=https://xxxx.ngrok-free.app
```

## 9. 常见问题（开发）

| 问题 | 排查 |
|------|------|
| AI 超时 / 无响应 | Dify 是否运行；`.env` 中 `DIFY_BASE_URL` 与 API Key |
| 容器访问不到 Dify | 使用 `host.docker.internal`，勿用 `localhost` |
| 前端仍显示 Mock | `VITE_USE_MOCK=false` 并重启 `npm run dev` |
| JWT 校验失败 | 各服务 `JWT_SECRET` 必须相同 |
| MySQL 连接失败 | 等待 healthcheck（首次约 30s）；核对密码 |
| Embedding 维度错误 | `MILVUS_DIMENSION` 须与模型输出一致（qwen3-embedding:0.6b = 1024） |

更多业务向 FAQ 见 [使用说明.md](./使用说明.md#5-常见问题)。
